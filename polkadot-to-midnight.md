# From Polkadot to Midnight — Developer Field Guide

*You already think in Rust, runtime upgrades, and parachain architecture. Time to think in ZK proofs and private witnesses.*

**Midnight Network Foundation | June 2026**

> **Version note:** Midnight mainnet launched March 31, 2026 in a federated phase — a curated set of trusted node operators (IOG, Google Cloud, Vodafone) runs the network while decentralization is completed via Cardano SPO onboarding. Compact targets `language_version >= 0.23`. Cross-contract calls are reserved in the language spec but not yet implemented in Compact 1.0. Always verify against [docs.midnight.network](https://docs.midnight.network) before shipping.

## 1. Why Midnight? The Pitch for Polkadot Developers

Polkadot's core thesis is that blockchains should be interoperable and appchain-specialized. You've internalized that thesis — whether you're writing ink! contracts, building Substrate pallets, or architecting parachains. You understand that the runtime is the product, that WebAssembly is the right compilation target, and that cross-chain communication is a first-class concern.

What Polkadot doesn't give you — at least not at the smart contract layer — is privacy. Every ink! contract's storage is public. Every pallet's state is readable by anyone with an RPC connection. Every transaction is visible on-chain. For regulated financial applications, enterprise data sharing, confidential business logic, or compliance-driven workflows, this is a blocker.

**One-line pitch:** Midnight is purpose-built for the cases where you need programmable on-chain logic but cannot expose the underlying data. It adds privacy at the protocol level using zero-knowledge proofs — not a workaround, not an off-chain hack, not a trusted enclave. ZK proofs are the execution model.

### The ink! Situation

Before anything else: **Parity stopped actively maintaining ink! in January 2026.** Issues and pull requests are locked down on the GitHub repository. ink! exists in community hands now, with an uncertain maintenance trajectory. If you're building new contracts in the Polkadot ecosystem today, this context matters — and it makes Midnight a more natural migration target than it was a year ago. Midnight's Compact language has a similar TypeScript-flavored syntax philosophy and a more actively invested maintainer in the Midnight Foundation.

### What Carries Over from Polkadot

- **Substrate is the actual foundation.** Midnight is built on the Polkadot SDK (formerly Substrate), using AURA for block production and GRANDPA for finality, extended with Cardano partner chain components for validator selection via Cardano SPOs. Your node operation, runtime, and infrastructure intuitions are directly relevant.
- **Rust type-safety instincts transfer.** Compact is TypeScript-like, not Rust — but your discipline around types, bounds, and compile-time correctness maps cleanly.
- **Determinism orientation.** Midnight contracts, like Substrate runtimes, produce the same result given the same inputs. Predictability before submission is a shared property.
- **TypeScript off-chain tooling.** Midnight.js SDK is TypeScript, familiar to polkadot.js / PAPI users.
- **Privacy-aware design thinking.** Polkadot developers already reason about parachain-level data boundaries and XCM trust assumptions.
- **Compile-time correctness orientation.** Both Rust and Compact compilers are strict and opinionated — Compact will feel familiar in that respect.

### What Needs Rewiring

- **ink! / Rust → Compact:** new language, new compiler target (ZK circuits, not Wasm)
- **UTXO-primary state model:** Midnight's primary model for tokens is UTXO-based via Zswap; account-style tokens are also possible via Compact contract storage
- **Pallets → no equivalent:** Midnight has no pallet/parachain layer; circuits are the only primitive
- **XCM → no direct equivalent yet:** Midnight cross-chain interoperability is a roadmap item (LayerZero integration targeted for Q3 2026 "Hua" phase)
- **Weight-based fees → DUST-based fees:** non-transferable resource generated by holding NIGHT
- **`Self::env().caller()` → does not exist:** access control is ZK-proved, not caller-identity-based
- **On-chain Wasm execution → off-chain ZK proof generation:** validators verify proofs, not re-execute logic
- **Storage deposits → no equivalent:** Midnight has no per-byte storage deposit mechanism
- **Runtime upgrades → redeploy:** no forkless in-place upgrades; state migration is your problem

## 2. Architecture: Polkadot's Stack vs. Midnight's Model

### Polkadot's Three-Layer Stack

- **Relay Chain:** Provides shared security and consensus coordination. Not where smart contracts live.
- **Parachains:** Application-specific blockchains with custom Substrate runtimes. Each parachain implements its own state transition function via pallets. The parachain IS the product.
- **Smart contracts (ink!):** Code deployed into a parachain that includes `pallet-contracts`. Compiled to WebAssembly. Executed by the Wasm VM inside the parachain runtime.

> Polkadot developers often frame the choice as: "Do I need a full parachain slot, or is a smart contract sufficient?" Midnight doesn't have this layered choice. It's a single-chain environment where your Compact contract is the primary building block — closer in shape to deploying an ink! contract than building a full parachain runtime.

### Midnight's Simpler Stack

Midnight is not a relay chain and not a parachain framework. It is itself a Substrate-based chain — built on the Polkadot SDK, using AURA for block production and GRANDPA for finality, extended with Cardano partner chain components for validator selection. Polkadot developers are working with genuinely familiar infrastructure at the node and runtime level. What differs is the smart contract layer: Compact contracts with ZK proofs replace ink! contracts with Wasm execution, and the privacy model inverts the transparency defaults you're used to.

| Concept | Polkadot | Midnight |
|---|---|---|
| Base infrastructure | Relay Chain (shared security) | Polkadot SDK — AURA + GRANDPA |
| Application layer | Parachains (custom Substrate runtimes) | No parachain concept |
| Smart contract layer | ink! on `pallet-contracts` parachain | Compact contracts deployed directly |
| Execution environment | WebAssembly VM in parachain runtime | ZK circuits run off-chain; proof on-chain |
| Runtime upgrades | Forkless via on-chain governance | Redeploy new contract; migrate state explicitly |
| Cross-chain messaging | XCM | No native equivalent (Q3 2026 roadmap) |
| Parachain slots | Coretime purchase | Not applicable |
| Custom pallets | FRAME pallets in Substrate runtime | Not applicable; use Compact circuits |
| Validator set | Relay Chain NPoS | Cardano SPOs + optional permissioned validators |
| Mainnet status | Fully decentralized | Federated phase (trusted operators); decentralization in progress |

### The Execution Model Inversion

This is the single biggest conceptual departure. In Polkadot's ink! model, your Wasm contract is deployed to the chain and executed on-chain by the Wasm VM inside the parachain runtime. Every validator re-executes your contract to reach consensus on the result. Everything is transparent because consensus requires it.

In Midnight, your Compact contract compiles to a ZK circuit. When a user invokes a circuit, it runs locally on their machine. Private data (spending keys, off-chain records) are consumed as private inputs called witnesses. The result is a cryptographic proof — submitted to the chain. Validators verify the proof, not re-execute the logic. Private inputs are never on-chain.

| Step | Polkadot ink! | Midnight Compact |
|---|---|---|
| Contract deployed as | Wasm bytecode on-chain | ZK circuit definition + ledger state |
| User invokes | Message call via extrinsic | Exported circuit via transaction |
| Computation runs | On every validator (on-chain Wasm VM) | Locally on user's machine (off-chain) |
| Private inputs | Not possible — all inputs visible | Witness functions — local, never on-chain |
| Chain receives | Signed extrinsic with call data | Transaction + ZK proof of correctness |
| Validators do | Re-execute the Wasm contract | Verify the proof only (faster) |
| State update | Contract storage mutated in-place | Ledger state updated; UTXOs created |

> **Proof generation is the new UX-critical latency.** On Polkadot, your user waits for block inclusion. On Midnight, they also wait for proof generation — which happens client-side via a **proof server daemon** (a separate process you run alongside your dApp) and currently takes seconds for typical circuits. Unlike weight-based gas limits, you optimize for constraint count at compile time, not runtime cost estimation.

## 3. ink! vs. Compact: The Language Comparison

ink! is an embedded DSL that extends Rust — you write Rust, add ink! macros, and get smart contracts. Compact is a standalone TypeScript-like language compiled to ZK arithmetic circuits. The targets are completely different: ink! → WebAssembly, Compact → ZK circuits.

### Anatomy of an ink! Contract

```rust
#[ink::contract]
mod my_contract {
    #[ink(storage)]
    pub struct MyContract {
        owner: AccountId,
        value: u64,
    }

    impl MyContract {
        #[ink(constructor)]
        pub fn new(initial_value: u64) -> Self {
            Self {
                owner: Self::env().caller(),   // caller identity available
                value: initial_value,
            }
        }

        #[ink(message)]
        pub fn set_value(&mut self, new_value: u64) {
            assert_eq!(self.env().caller(), self.owner, "Not authorized");
            self.value = new_value;
            self.env().emit_event(ValueChanged { new_value });
        }

        #[ink(message)]
        pub fn get_value(&self) -> u64 { self.value }
    }

    #[ink(event)]
    pub struct ValueChanged { new_value: u64 }
}
```

### Anatomy of a Compact Contract

```compact
pragma language_version >= 0.23;
import CompactStandardLibrary;

// On-chain state — each field declared individually (not a block)
export ledger owner: Bytes<32>;    // stored commitment of the owner's key
export ledger value: Uint<64>;
export ledger round: Counter;      // anonymity counter — always include

// Witness: private key from user's wallet — never leaves their machine
witness getSpendingKey(): Bytes<32>;

constructor(ownerKey: Bytes<32>) {
    owner = disclose(ownerKey);    // store commitment at deployment
    round.increment(1);
}

// Circuit: equivalent to #[ink(message)] — callable in a transaction
export circuit setValue(newValue: Uint<64>): [] {
    // No Self::env().caller() — prove key ownership via ZK instead
    assert(persistentHash(getSpendingKey()) == owner, "Not authorized");
    value = disclose(newValue);
    round.increment(1);
    // No events — state change is observable via ledger fields
}

export circuit getValue(): Uint<64> {
    return value;   // public ledger field — no disclose() needed here
}
```

**Ledger syntax note:** Each ledger field is declared with its own `export ledger fieldName: Type;` statement. There is no `ledger { }` block syntax in current Compact — that appeared in pre-mainnet versions and will fail to compile.

**`getValue()` note:** A read-only circuit returning a public ledger field does not need `disclose()`. The value is already public. `disclose()` is only required when private-origin data (from a witness) crosses into the public domain.

### ink! vs. Compact: Concept Mapping

| ink! / Substrate | Compact / Midnight | Notes |
|---|---|---|
| `#[ink::contract] mod` | File IS the contract | One `.compact` file = one contract |
| `#[ink(storage)] struct` | `export ledger field: Type;` | On-chain state; typed, not raw bytes |
| `#[ink(constructor)]` | `constructor()` | Same concept |
| `#[ink(message)]` | `export circuit myFn(): RetType` | `export` = callable in a transaction |
| `#[ink(message, payable)]` | No equivalent | No inline token send; use Zswap flows |
| `Self::env().caller()` | No equivalent | Caller proved via `persistentHash(witness())`, not exposed |
| `Self::env().transferred_value()` | No equivalent | No inline value transfer |
| `Self::env().block_timestamp()` | `round: Counter` (ledger field) | No timestamps; use round counter for ordering |
| `self.env().emit_event(E{})` | No events | State changes are observable; no separate event log |
| `assert_eq!(a, b, "msg")` | `assert(a == b, "msg")` | Both abort execution on failure |
| `panic!("msg")` | `assert(false, "msg")` | Proof fails; transaction rejected |
| cross-contract `CallBuilder` | **Not implemented in Compact 1.0** | `contract` keyword reserved, not functional |
| `ink::Mapping<K, V>` | Standard library collections | Compact has bounded `Map`, `MerkleTree` types |
| `ink::StorageVec` | Compact vector types (bounded) | No dynamic allocation |
| `cargo-contract` | `compactc` | The Compact compiler CLI |
| `polkadot.js` / PAPI | Midnight.js SDK | TypeScript off-chain SDK |
| `substrate-contracts-node` | Midnight DevNet | Local chain for development |
| `.contract` artifact | Compiler-generated TypeScript stubs | Typed interfaces to your circuits, auto-generated |
| Storage deposit | None | No storage deposit mechanism |
| Weight (`ref_time`, `proof_size`) | Circuit constraint count | Cost model is compile-time, not runtime weight |

### A Note on Cross-Contract Calls

This deserves its own callout because it affects architecture significantly.

In ink!, `CallBuilder` is a foundational primitive — PSP22 approvals, governance contracts, DEX routers, and virtually any composable protocol depend on contracts calling other contracts at runtime.

**In Compact 1.0, this does not exist.** The `contract` keyword is reserved in the language spec for future cross-contract call support, but it is not currently implemented. Two separately deployed contracts cannot call each other at runtime.

The current workaround: **compile-time module imports.** You can import and reuse Compact modules from libraries (like `@openzeppelin/compact-contracts`) within a single compiled contract. Shared logic is composed at compile time, not at runtime. This means what would be a two-contract system in ink! may need to be a single larger Compact contract — design accordingly. OpenZeppelin's own `Ownable` module explicitly notes that contract-to-contract ownership transfer is unsupported until cross-contract calls land.

## 4. State Model: From Public Storage to Private Ledger

In ink!, your `#[ink(storage)]` struct is global, mutable, and entirely public. Anyone can read any storage slot from any ink! contract via RPC. There is no hiding mechanism at the storage layer.

Midnight's privacy model inverts this. The compiler tracks data flow. Any value derived from a witness function is considered private. You cannot accidentally write private data to the ledger — the compiler rejects it unless you explicitly annotate it with `disclose()`. This is a compile-time privacy guarantee, not a runtime check.

### The `disclose()` Annotation: Intentional Disclosure

```compact
witness _privateBalance(): Uint<64>;  // runs locally; never on-chain

export circuit updateTotal(): [] {
    const bal = _privateBalance();    // private origin

    // ❌ COMPILE ERROR: witness-derived value cannot land on-chain directly
    ledger.total = ledger.total + bal;

    // ✅ CORRECT: explicitly cross the privacy boundary
    ledger.total = ledger.total + disclose(bal);
}
```

`disclose()` is a compiler annotation, not a cast. It signals: "I know this value is leaving the private domain; I accept responsibility." Think of it as a mandatory audit trail for every privacy boundary crossing. The value only appears on-chain if actually stored in ledger state or returned from a circuit.

### Three Data Domains vs. ink!'s One

| Domain | Where it lives | Who can read it | ink! equivalent |
|---|---|---|---|
| Witness data | User's local machine only | Only the user | Not possible in ink! |
| Private circuit execution | In-flight during proof generation | Nobody | Not possible in ink! |
| Disclosed ledger state | Public ledger on-chain | Everyone | Contract storage (the entire ink! storage model) |

### Shielded vs. Unshielded UTXOs

Token UTXOs in Midnight can be shielded or unshielded — a per-UTXO choice:

- **Unshielded:** Value and owner visible on-chain. Same transparency as reading an ink! storage mapping.
- **Shielded:** Value and owner hidden behind cryptographic commitments. The network sees a commitment hash; only the owner knows the contents.

A single transaction can mix both. A regulated asset might be shielded for user-to-user transfers but disclosed to a compliance auditor via viewing key — in the same transaction.

### No Storage Deposits

`pallet-contracts` requires storage deposits: when a contract allocates storage, the caller pays a deposit proportional to bytes used. Returned when storage is freed. This prevents ledger bloat and ensures storage has an economic cost.

Midnight has no storage deposit mechanism. Ledger state declared in `export ledger` fields is allocated at deployment. No per-byte fee for storage updates, no deposit to track, no refund logic to implement. This removes a significant class of economic footguns from contract design.

## 5. Access Control Without Caller Identity

`Self::env().caller()` is the cornerstone of ink! access control. It's how you implement `Ownable`, role-based access, and multi-sig patterns. On Midnight, **it does not exist** — and this is deliberate.

Exposing the caller's identity on-chain would undermine Midnight's privacy model. If the chain knew who called every circuit, transaction graphs would leak identities. Instead, Midnight uses ZK-proved knowledge of a secret as the authorization mechanism.

### The Canonical Access Control Pattern

Store a commitment to the owner's key in ledger state at deployment. In any restricted circuit, use a witness to retrieve the caller's private key locally, hash it, and assert it matches the stored commitment. The ZK proof verifies this without the private key ever touching the chain.

```rust
// ink! pattern
#[ink(storage)]
pub struct MyContract {
    owner: AccountId,   // caller identity stored at deploy
}
// In a message:
assert_eq!(self.env().caller(), self.owner, "Not authorized");
```

```compact
// Compact equivalent
export ledger owner: Bytes<32>;   // commitment stored at deploy

constructor(ownerCommitment: Bytes<32>) {
    owner = disclose(ownerCommitment);
}

witness getSpendingKey(): Bytes<32>;

export circuit restrictedAction(param: Uint<64>): [] {
    // Prove caller knows the private key behind owner commitment
    assert(persistentHash(getSpendingKey()) == owner, "Not authorized");
    value = disclose(param);
    round.increment(1);
}
```

> **Every restricted circuit needs its own explicit `assert`.** There is no inherited authorization context between circuits — no equivalent of Rust's method visibility or a shared modifier. If you add a new exported circuit and forget the `assert`, it's callable by anyone. Make access control the first line of every restricted circuit. Build the habit.

### Multi-Role Access Control

ink! multi-role patterns typically use a `Mapping<AccountId, Role>`. The Compact equivalent stores a commitment per authorized role in separate ledger fields:

```compact
export ledger admin: Bytes<32>;
export ledger complianceOfficer: Bytes<32>;
export ledger round: Counter;

witness getSpendingKey(): Bytes<32>;

export circuit adminAction(param: Uint<64>): [] {
    assert(persistentHash(getSpendingKey()) == admin, "Not admin");
    // ... admin logic
    round.increment(1);
}

export circuit complianceAction(param: Uint<64>): [] {
    assert(persistentHash(getSpendingKey()) == complianceOfficer, "Not compliance officer");
    // ... compliance logic
    round.increment(1);
}
```

## 6. Tokens, Fees, and the DOT → NIGHT/DUST Shift

### NIGHT and DUST: Two Tokens, One Purpose Each

**NIGHT** is the transferable utility token. Used for staking (network security), governance, and passive DUST generation. Closest analogy to DOT — but unlike DOT, NIGHT is not spent directly as a fee medium. All NIGHT transfers are publicly visible on-chain. It is deliberately not a privacy coin.

**DUST** is a non-transferable resource generated by holding NIGHT. Used exclusively to pay transaction fees. Expires if unused. Non-market-priced — you generate it by holding NIGHT, not by buying it.

> **Analogy:** Polkadot's weight-based fees are paid in DOT or the parachain's native token. Fee spikes under congestion are a known operational concern. Midnight's DUST fees decouple transaction costs from NIGHT's market price entirely. You generate DUST passively — it's a usage allowance, not a price you pay. The more NIGHT you hold, the faster DUST refills.

**Key DUST properties:**
- **Shielded.** Fee payment doesn't expose your wallet address.
- **Non-transferable.** Cannot be sent between wallets — but can be *delegated* to sponsor others' transactions.
- **Decaying.** Unused DUST bleeds off; if you move your NIGHT or un-designate the generating address, it decays to zero.
- **Renewable.** Consume it on a transaction and it refills over time based on NIGHT holdings.

**The cold-start problem.** New wallets have no DUST. Without DUST, they cannot pay fees. The solution is **DUST sponsorship**: a dApp holding NIGHT delegates its DUST to cover new users' fees. The Midnight SDK exposes this as a two-phase transaction balancing model — the user covers token flows, the sponsor covers DUST. Design sponsored transactions into your dApp from day one if you expect users who don't hold NIGHT.

### Weight vs. Circuit Constraints: The Fee Model Comparison

| Aspect | Polkadot / ink! | Midnight |
|---|---|---|
| Fee token | DOT or parachain native | DUST (non-transferable, generated by NIGHT) |
| Fee model | Weight: `ref_time` + `proof_size` | DUST consumed per transaction |
| Cost estimation | Simulate + dry-run for weight | Circuit constraint count (compile-time) |
| Fee market | Tip for priority inclusion | Dynamic based on network load; no competitive mempool |
| Paid by | Extrinsic payer | DUST from user's wallet (or sponsored) |
| Failed tx cost | Base fee still charged | Guaranteed-phase DUST forfeited on fallible failure |
| Storage cost | Storage deposit (refundable) | No storage deposit mechanism |
| Payable messages | `#[ink(message, payable)]` | Not available; use Zswap token flows |

### PSP22 vs. Midnight Token Models

Midnight supports two token models — the right one depends on your use case:

**UTXO-based tokens (via Zswap):** Tokens live as discrete unspent outputs on the ledger. Shielding is native. Atomic multi-asset swaps are protocol-level. The primary model for private transfers and DeFi.

**Account-style tokens (via Compact contract storage):** Ledger fields track balances. Useful for governance tokens with delegation, complex DeFi state machines, gaming systems where account composability is more natural than UTXO flows.

| Property | PSP22 (ink!) | Midnight UTXO tokens | Midnight account-style tokens |
|---|---|---|---|
| Balance storage | `Mapping<AccountId, Balance>` | UTXOs on the ledger | `export ledger` field in Compact contract |
| Transfer | Storage mutation: from -= x, to += x | Consume UTXO, create UTXO with ZK proof | Circuit mutates ledger state |
| Token identity | Contract address | Hash(contractAddr, domainSeparator) | Defined by the contract |
| Spend authorization | Caller check via `env().caller()` | ZK proof of UTXO ownership | ZK proof of key knowledge |
| Privacy | None — all balances public | Shielded UTXOs hide value and owner | Public by default; `disclose()` controls |
| Atomic swap | Multi-message / cross-contract | Native via Zswap balance vector | Circuit-level composition |
| Best for | Most PSP22/ERC-20 equivalent cases | Private transfers, privacy-first DeFi | Complex governance, gaming |

### Zswap: Atomic Multi-Asset Swaps

Zswap is Midnight's native atomic swap protocol, built into the transaction layer — not a program or pallet on top. The core concept is an **offer**: a bundle specifying input UTXOs consumed and output UTXOs created, with a balance vector tracking net value flow per token type. Two parties' offers can be merged non-interactively by an off-chain matcher. The matcher verifies that the combined balance vector nets to zero without seeing the amounts.

```
// Conceptual Zswap offer structure (pseudocode)
AliceOffer {
    inputs:  [shielded_NIGHT_100],       // Alice spends 100 NIGHT
    outputs: [shielded_USDC_95],         // Alice receives 95 USDC
    balance: { NIGHT: -100, USDC: +95 }
}
BobOffer {
    inputs:  [shielded_USDC_95],         // Bob spends 95 USDC
    outputs: [shielded_NIGHT_100],       // Bob receives 100 NIGHT
    balance: { NIGHT: +100, USDC: -95 }
}
// Combined balance = { NIGHT: 0, USDC: 0 } ✓ — swap is complete
// Neither party's amounts are visible to the matcher
```

No `approve()` pattern. No front-running exposure. Token type identity is cryptographically derived from the issuing contract's address — no global registry needed.

> **Zswap implementation note:** The official docs flag this layer as not yet performance-optimized and subject to further revision. The conceptual model is stable; verify specific API surface and proof costs against current docs before shipping production integrations.

### XCM and Cross-Chain: What's Missing

XCM is one of Polkadot's most distinctive features. Midnight currently has no equivalent. Cross-chain communication from Midnight to other chains is a roadmap item — LayerZero integration is targeted for the Q3 2026 "Hua" phase, which would let Midnight function as privacy infrastructure for any chain connecting through LayerZero. That's not live today.

If your Polkadot application relies on XCM for multi-chain token movement, cross-chain contract calls, or governance messaging, that functionality will not port to Midnight in its current form. Factor this into your architecture early. Midnight is best suited today for self-contained applications where the privacy layer is the primary value.

## 7. Circuits and Witnesses: The ZK Execution Model

ink! developers write Rust functions that the Wasm VM executes deterministically on-chain. Compact developers write circuits — functions with fixed computational bounds that compile to arithmetic circuits for ZK proof generation. The constraints shape how you architect logic.

### What a Circuit Can and Cannot Do

| Capability | ink! message (Wasm) | Compact circuit (ZK) |
|---|---|---|
| Bounded loops | Yes (runtime-metered by weight) | Yes, but bounds must be compile-time constants |
| Dynamic allocation | Yes (heap via `Vec`, etc.) | No — all sizes known at compile time |
| Cross-contract calls | Yes (`CallBuilder`) | **Not yet implemented in Compact 1.0** |
| Ledger read/write | Direct (`self.field`) | Via `export ledger` fields |
| Block timestamp | `Self::env().block_timestamp()` | Not available — use `round` counter |
| Caller identity | `Self::env().caller()` | Not available — use `persistentHash(witness())` |
| Random numbers | `Self::env().random()` (deprecated) | Not available natively |
| Event emission | `self.env().emit_event()` | No events — observe ledger state changes |
| Recursion | Yes (bounded by weight) | No recursive circuits |

### Witness Functions: ink! Has No Equivalent

ink! contracts have no concept of private inputs. Every argument to every message is visible in the extrinsic. The contract's storage is public. There is no mechanism for a user to supply data to a contract without that data being on-chain.

Compact's witness functions are the key innovation. A witness runs locally on the user's machine before proof generation. It can access the user's private key, query an off-chain database, read local state — anything. Its return values feed into the circuit as private inputs never submitted to the chain.

```compact
// Witnesses: run locally, access private data
// Implementations provided in TypeScript by the dApp developer
witness getSpendingKey(): Bytes<32>;
witness getPrivateScore(userId: Bytes<32>): Uint<64>;

export circuit submitScore(): [] {
    const sk    = getSpendingKey();          // private — stays local
    const score = getPrivateScore(owner);    // private — from off-chain DB

    // Verify authorization
    assert(persistentHash(sk) == owner, "Not authorized");

    // Verify score in valid range without revealing the actual value
    assert(score >= 0 && score <= 1000, "Score out of range");

    // Only the range-validity fact hits the chain; actual score stays private
    ledger.scoreSubmitted = disclose(true);
    round.increment(1);
}
```

This circuit proves the user's score is in range 0–1000 without revealing the actual score. The chain stores only the boolean result. This is the pattern that enables compliance reporting — prove you meet a threshold without full disclosure. It's the core use case Midnight is designed for.

> **Witness security rule:** Witness implementations run off-chain and are **not cryptographically verified** by the circuit compiler. Each user supplies their own. Contract logic must validate witness-derived values in-circuit before acting on them — treat every witness result as untrusted user input. The `@openzeppelin/compact-contracts` library ships test witnesses only, not production witnesses, for exactly this reason. If you use it, author and audit your own witnesses.

## 8. Transactions and the Two-Phase Execution Model

This is absent from most Midnight documentation and from the ink! mental model — read carefully.

### What a Midnight Transaction Contains

- One or more circuit invocations with their ZK proofs
- UTXO inputs being consumed (shielded or unshielded)
- UTXO outputs being created (payments, change, new token UTXOs)
- Nullifiers for spent UTXOs (prevent double-spend)
- DUST fee payment

### The Two-Phase Execution Model

ink! extrinsics are all-or-nothing: they fully succeed or the entire transaction reverts (though weight fees are still charged). Midnight splits execution into three stages with importantly different semantics:

**Stage 1 — Well-formedness check:** Runs against no state. Verifies all ZK proofs and that offers are balanced (accounting for fees and mints). A bad proof or unbalanced offer → rejected outright, not included in the ledger.

**Stage 2 — Guaranteed phase:** Runs against ledger state. **DUST fees are collected here.** If this phase fails, the transaction is *not included at all* — similar to a pre-dispatch check failing in Substrate.

**Stage 3 — Fallible phase:** Also runs against ledger state. If this fails, the effects of the guaranteed phase **still apply**, and the transaction is recorded as a **partial success**. Fees are forfeited.

**Why this matters:** Unlike ink!'s clean revert, a Midnight transaction can land in a partially-applied state. Design your circuits so that a partial success is always safe. Put DUST payment and must-succeed invariants in the guaranteed phase. Put speculative or best-effort effects in the fallible phase — and never put a step there that a guaranteed-phase commitment already assumed would complete.

**On fees for failed transactions:** If the transaction fails in the well-formedness check, it's not included and no DUST is spent. If it fails in the guaranteed phase, it's not included. If it fails in the fallible phase, guaranteed-phase DUST is forfeited. The "check current docs" placeholder is answered by the two-phase model.

## 9. Pallets vs. Circuits: Rethinking What Goes Where

Experienced Polkadot developers have a decision framework: pallet (runtime module) or smart contract? Pallets get privileged runtime access, lower overhead, and can modify the chain's state transition function. Smart contracts are sandboxed but faster to develop and deploy.

Midnight eliminates this distinction entirely. There are no pallets in Midnight. All programmable logic is expressed as Compact circuits. The circuit IS the primitive.

| Pattern | Polkadot approach | Midnight approach |
|---|---|---|
| Token issuance | `pallet-assets` or PSP22 contract | Compact circuit + Zswap balance vector |
| Access control | `pallet-membership` or ink! Ownable | ZK-proved key commitment in circuits |
| Governance | `pallet-democracy` / OpenGov | Circuit-based voting (custom implementation) |
| Staking logic | `pallet-staking` | Protocol-level (NIGHT staking is built-in) |
| Upgradeable logic | Runtime upgrade via governance | Redeploy + explicit state migration |
| Cross-chain transfer | XCM + `pallet-xcm` | Not available natively (Q3 2026 roadmap) |
| Identity / credentialing | `pallet-identity` | ZK credential circuits (custom) |
| Oracle data | Off-chain workers + pallet | Witness functions (off-chain data injection) |
| Batch operations | `utility.batch` extrinsic | Bundle multiple circuit calls in one transaction |

### Runtime Upgrades: The Biggest Operational Difference

Polkadot's forkless runtime upgrades are a celebrated feature — the chain's state transition function can be replaced entirely via on-chain governance. No hard fork, no coordinated node restart. This is enabled by storing the Wasm runtime blob on-chain and having every validator execute the current on-chain runtime.

Midnight contracts don't upgrade this way. Compact contracts are deployed once. To change logic, you deploy a new contract. State migration must be handled explicitly.

For Polkadot developers who've relied on runtime upgrades to fix production bugs, this requires a shift in approach:

- Design contracts defensively from the start
- Test exhaustively before deployment — treat every deployment as potentially final
- Plan state migration logic for any contract that might need future logic changes
- Consider proxy-style patterns (new contract reads state from the old one during a migration window) for critical systems

## 10. Toolchain and Development Environment

Polkadot's toolchain is Rust-centric and deep: `cargo-contract` for ink!, `substrate-contracts-node` for local testing, polkadot.js / PAPI for off-chain interaction. Midnight's toolchain is TypeScript-centric at the off-chain layer with a dedicated Compact compiler for on-chain logic — and a proof server daemon with no Polkadot equivalent.

### Tool Mapping: Polkadot → Midnight

| Polkadot Tool | Midnight Equivalent | Notes |
|---|---|---|
| ink! (Rust DSL) | Compact (`compactc`) | TypeScript-like; compiles to ZK circuits, not Wasm |
| `cargo-contract` | `compactc` CLI | Build, compile, deploy Compact contracts |
| `substrate-contracts-node` | Midnight DevNet | Local chain for development and testing |
| polkadot.js / PAPI | Midnight.js SDK | TypeScript SDK for tx building and submission |
| `.contract` artifact | Compiler-generated TypeScript stubs | Typed interfaces to your circuits |
| OpenBrush / ink! stdlib | `CompactStandardLibrary` + OZ Compact | Built-in stdlib; OZ library available (experimental) |
| `chain_extension` | No equivalent | Circuits are self-contained; no pallet-level hooks |
| ink! integration tests | Compact test suite | Tests run against local DevNet |
| Subscan / block explorer | Midnight block explorer | Public ledger state browsing |
| Kusama (canary network) | Midnight testnet | Deploy and test before mainnet |
| *(no equivalent)* | Proof server daemon | Separate process for client-side ZK proof generation |
| Midnight MCP | AI-assisted dev tool | Live Compact compiler integration for AI editors |

### Project Structure

```
my-midnight-dapp/
├── contracts/
│   ├── my_contract.compact          # Compact source (replaces src/lib.rs)
│   └── managed/                     # Compiler output — do not edit
│       └── my_contract/
│           ├── contract/index.cjs   # Generated TypeScript stubs (your typed ABI)
│           ├── keys/                # ZK proving/verifying keys
│           └── zkir/                # ZK intermediate representation
└── src/
    ├── index.ts                     # dApp logic via Midnight.js SDK
    └── witnesses.ts                 # Implement witness functions here
                                     # (spending key, off-chain data access)
```

### Development Workflow

1. **Write contract:** Author your `.compact` file. Declare ledger fields, witness function signatures, and exported circuits.
2. **Compile:** Run `compactc`. The compiler is strict: `disclose()` violations and non-constant loop bounds are compile errors, not runtime warnings.
3. **Start proof server:** Run the proof server daemon. Your dApp's SDK calls route through it for proof generation. This is a separate process from your dApp — budget for it in development setup and deployment architecture.
4. **Implement witnesses:** The compiler generates TypeScript interfaces for each witness function. You implement the body — spending key retrieval, off-chain data queries, etc.
5. **Test:** Run against a local Midnight DevNet. Test circuit correctness, state transitions, proof generation, and both happy paths and failing assertions.
6. **Deploy:** Use Midnight.js SDK. The deployer's wallet pays DUST. The contract is immutable post-deployment.
7. **Build dApp:** Use compiler-generated TypeScript stubs as typed circuit interfaces. Submit transactions via the SDK.

## 11. Security: What Changes, What Disappears, What's New

ink! has a well-documented set of pitfalls: missing caller checks, reentrancy through cross-contract calls, integer overflow (less common in Rust), storage deposit edge cases, and mishandling of transferred value. Midnight eliminates some structurally while introducing a small, distinct set of new concerns.

### ink! Vulnerabilities That Don't Apply on Midnight

- **Reentrancy via cross-contract calls:** Compact circuits are closed computations with no external calls mid-execution. No reentrancy vector exists.
- **Missing caller checks:** Replaced structurally — there is no caller identity to forget checking. If you omit the `assert(persistentHash(witness()) == owner)` pattern, the circuit simply has no access control, which is visible.
- **Storage deposit accounting errors:** No storage deposit mechanism; this entire class doesn't apply.
- **Payable message misuse:** No payable messages; no inline token transfer to mishandle.
- **Front-running via mempool:** Shielded transactions don't expose amounts or intent pre-broadcast. No MEV surface for value extraction.

### Security Patterns That Still Apply

- **`assert()` for invariants:** Same role as ink!'s `assert_eq!()`. Proof fails on violation — equivalent to a panicking extrinsic.
- **Arithmetic overflow:** Compact uses fixed-width integers (`Uint<64>`, `Uint<128>`). Rust's overflow protections don't exist here — audit all arithmetic carefully.
- **Access control completeness:** Every restricted circuit needs its own `assert`. No inherited context, no global modifier. Review every `export circuit`.
- **Round counter discipline:** Always include a `round: Counter` field in ledger state and increment it in every circuit that modifies state. This prevents transaction correlation and replay patterns. Use the stdlib `Counter` type — it's one line.

### New Attack Surfaces

**Disclosure mistakes** are the most common Compact bug class. Using `disclose()` on data you didn't mean to make public, or storing private-derived values in ledger state without realizing what you've exposed. The compiler catches accidental leakage of witness data — but can't catch *intentional* disclosure you didn't think through. Audit every `disclose()` call.

**Commitment brute-forcing:** If you commit to a small-domain value (a boolean, a 4-bit integer) without a random salt, the commitment is trivially invertible. Use `persistentCommit()` with randomness for small-domain secrets.

**Witness implementation bugs:** The Compact compiler validates the circuit logic and data flow. It cannot validate your TypeScript witness implementations. A witness that leaks data, returns stale values, or incorrectly handles errors will not be caught at compile time. Test witness logic independently and treat it as security-critical code.

**Untrusted witness values:** Witness implementations are provided by the executing user and run off-chain. Contract logic must never trust a witness value without validating it in-circuit. Treat every witness result as untrusted user input.

**Partial-success states:** Given the two-phase execution model (§8), a fallible-phase failure can leave your contract in a partially-applied state with no clean revert. Design circuit boundaries so partial success is always a safe state to be in.

**Circuit constraint complexity:** Overly complex circuits produce long proof times, degrading UX. Keep circuits lean and bounded. A crafted input that causes disproportionate proving time is a liveness concern if not a direct attack vector.

## 12. Quick Reference: ink! / Polkadot → Compact / Midnight

| ink! / Polkadot | Compact / Midnight | Notes |
|---|---|---|
| `#[ink::contract] mod` | File IS the contract | One `.compact` file = one contract |
| `#[ink(storage)] struct` | `export ledger field: Type;` | On-chain state; typed, not raw bytes |
| `#[ink(constructor)]` | `constructor()` | Same concept |
| `#[ink(message)]` | `export circuit myFn(args): RetType` | `export` = callable in a transaction |
| `#[ink(message, payable)]` | No equivalent | No inline value transfer |
| `Self::env().caller()` | `persistentHash(getSpendingKey())` | ZK proof replaces caller identity |
| `Self::env().transferred_value()` | No equivalent | No inline token send pattern |
| `Self::env().block_timestamp()` | `round: Counter` (ledger field) | No timestamps; use round for ordering |
| `self.env().emit_event(E{})` | No events | Index ledger state changes instead |
| `assert_eq!(a, b, "msg")` | `assert(a == b, "msg")` | Proof fails on violation |
| `panic!("reason")` | `assert(false, "reason")` | Transaction rejected |
| cross-contract `CallBuilder` | **Not implemented in Compact 1.0** | `contract` keyword reserved, not functional |
| `ink::Mapping<K, V>` | Compact stdlib `Map` (bounded) | Bounded types; compile-time size |
| `ink::StorageVec` | Compact vector types (bounded) | No dynamic allocation |
| Weight (`ref_time`, `proof_size`) | Circuit constraint count | Cost is compile-time, not runtime |
| Storage deposit | None | No deposit mechanism |
| `chain_extension` | No equivalent | No pallet-level hooks from circuits |
| Pallet (runtime module) | No equivalent | Circuits are the only primitive |
| Runtime upgrade via governance | Redeploy new contract | No forkless in-place upgrade |
| XCM cross-chain message | No equivalent (Q3 2026 roadmap) | Plan for absence in current designs |
| DOT / parachain token fees | DUST fees | Generated by holding NIGHT; non-traded |
| PSP22 token standard | Zswap UTXO tokens or account-style Compact tokens | Choose per use case |
| `cargo-contract` | `compactc` | The Compact compiler |
| `substrate-contracts-node` | Midnight DevNet | Local dev chain |
| polkadot.js / PAPI | Midnight.js SDK | TypeScript off-chain SDK |
| Kusama testnet | Midnight testnet | Test before mainnet |
| `u8` / `u64` / `u128` | `Uint<8>` / `Uint<64>` / `Uint<128>` | Fixed-width integers, explicit size |
| `AccountId` | `Bytes<32>` (public key commitment) | Addresses represented differently |
| `bool` | `Boolean` | Same concept |
| keccak256 hash | `transientHash()` / `persistentHash()` | ZK-optimized hash functions |
| Commitment with salt | `transientCommit(v, salt)` / `persistentCommit(v, rand)` | Use `persistent*` for ledger state |

## 13. Suggested Learning Path

1. **Install the toolchain** → start the proof server → clone `example-hello-world` → deploy to local DevNet. Get the compile → proving-keys → TypeScript-driver loop in your fingers before writing real logic. ([docs.midnight.network/getting-started](https://docs.midnight.network/getting-started))
2. **Work the bulletin-board tutorial.** It introduces witnesses, commitments, `disclose()`, the round-counter replay pattern, and the full compile → prove → submit flow in a realistic contract.
3. **Read the Compact language reference and standard library** ([docs.midnight.network/compact](https://docs.midnight.network/compact)). Pay attention to ledger types: `Counter`, `Map`, `MerkleTree`, `Maybe`.
4. **Read the "How Midnight Works" concept pages** — transaction semantics (especially guaranteed/fallible phases), Zswap, DUST architecture, Kachina. Protocol-level reasoning lives here.
5. **Build something with a native token** to feel the Zswap UTXO model, then add selective disclosure to feel the compliance angle.
6. **Explore `@openzeppelin/compact-contracts`** for `Ownable`, `Pausable`, and `FungibleToken` patterns — actively maintained but flagged experimental. Author your own witnesses; do not reuse the test witnesses shipped with the library.
