## 1. Why Midnight? The Pitch for Polkadot Developers

Polkadot's core thesis is that blockchains should be interoperable and appchain-specialized. You've internalized that thesis — whether you're writing ink! contracts, building Substrate pallets, or architecting parachains. You understand that the runtime is the product, that WebAssembly is the right compilation target, and that cross-chain communication is a first-class concern.

What Polkadot doesn't give you — at least not at the smart contract layer — is privacy. Every ink! contract's storage is public. Every pallet's state is readable by anyone with an RPC connection. Every transaction is visible on-chain. For regulated financial applications, enterprise data sharing, confidential business logic, or compliance-driven workflows, this is a blocker.

The Polkadot ecosystem went through a significant shift in early 2026. Polkadot Hub launched as the primary smart contract destination — a production-ready platform with a dual-VM architecture supporting both REVM (full EVM compatibility, deploy Solidity unmodified) and PVM (native RISC-V execution via `pallet-revive`). At the same time, Parity dropped active maintenance of ink! in January 2026. The ink! Alliance — a group of former Parity engineers — picked it up with Polkadot Treasury funding and continues development, but the ecosystem's center of gravity has shifted decisively toward Solidity-on-REVM as the primary smart contract path.

This context matters for a guide to Midnight because **the most likely Polkadot developer reading this is now a Solidity developer**, not a Rust/ink! developer. If that's you, the Ethereum guide in this series covers the Solidity→Midnight conceptual mapping in depth. This guide focuses on what's specific to the Polkadot ecosystem context: the Substrate infrastructure you're already running, the appchain vs. smart contract decision framework you're used to, and what Midnight offers that neither REVM nor PVM currently provide.

**One-line pitch:** Midnight is purpose-built for the cases where you need programmable on-chain logic but cannot expose the underlying data. It uses zero-knowledge proofs as the execution model — privacy is structural, not a layer you add. And because Midnight is itself built on the Polkadot SDK (Substrate), your node infrastructure intuitions transfer directly.

### What Carries Over from Polkadot

- **Substrate is the actual foundation.** Midnight is built on the Polkadot SDK using AURA for block production and GRANDPA for finality, extended with Cardano partner chain components for validator selection. Your node operation, runtime, and infrastructure instincts are directly relevant.
- **TypeScript off-chain tooling.** Midnight.js SDK is TypeScript — familiar to polkadot.js / PAPI users.
- **Compile-time correctness orientation.** Whether you write Solidity with strict settings, Rust with Anchor, or ink!, you're used to the compiler catching errors before deployment. Compact's compiler is similarly strict.
- **Privacy-aware design thinking.** Polkadot developers already reason about parachain-level data boundaries and cross-chain trust assumptions.

### What Needs Rewiring

- **Solidity (REVM) or Rust/ink! → Compact:** new language, new compiler target (ZK circuits, not EVM bytecode or Wasm)
- **UTXO-primary state model:** Midnight's token model is UTXO-based via Zswap; account-style tokens are possible via Compact contract storage
- **XCM → no direct equivalent yet:** Midnight cross-chain interoperability is a roadmap item
- **Weight-based fees → DUST-based fees:** non-transferable resource generated by holding NIGHT
- **No `msg.sender` / signer list:** access control is ZK-proved, not caller-identity-based
- **On-chain execution → off-chain ZK proof generation:** validators verify proofs, not re-execute logic
- **Runtime upgrades → redeploy:** no forkless in-place upgrades; state migration is your problem
- **Privacy is absent by default on Polkadot Hub (REVM/PVM):** on Midnight, privacy is structural

## 2. The Polkadot Hub Smart Contract Landscape (June 2026)

Before the Midnight comparison, it's worth being precise about what you're migrating *from* — because the answer changed significantly in 2025–2026.

### Polkadot Hub: Dual-VM Architecture

Polkadot Hub provides a production-ready smart contract platform with a dual-VM approach offering two execution backends: REVM for unmodified EVM compatibility and native PVM for optimized computationally expensive workloads.

**REVM (primary path for most developers):** Polkadot Hub enables developers to deploy and interact with Solidity contracts through REVM, a high-performance, Rust-based Ethereum Virtual Machine implementation. Ethereum-native developers can use Polkadot-native precompiles to access Polkadot features — such as governance, treasury, multisig, and XCM — within a unified, interoperable runtime environment. Full MetaMask, Hardhat, Foundry, Remix compatibility. Deploy Solidity contracts with zero modifications.

**PVM (performance-intensive workloads):** For projects that require maximum computational performance, Polkadot Hub also supports PolkaVM (PVM), a native RISC-V execution engine. PVM is optional and designed for high-throughput, performance-intensive smart contracts. You can write PVM contracts in Solidity (via `resolc`, the Solidity-to-PVM compiler) or Rust. PVM smart contracts with Ethereum compatibility are in early-stage development and may be unstable or incomplete.

**Revive:** The pallet included in the Polkadot Asset Hub runtime that provides a unified execution environment, enabling smart contracts written for both the Polkadot Virtual Machine (PVM) and the Ethereum Virtual Machine (EVM) to run within the same runtime.

### ink! Current Status

Since January 2026, Parity is unable to actively maintain or develop ink! further. However, ink! has become a community project maintained by the ink! Alliance, a loose group of former Parity employees and teams, funded by a Polkadot Treasury grant. It is not abandoned — it is community-maintained with active funding. The ink! Alliance continues releasing updates; ink! now targets PVM via `pallet-revive` rather than the older Wasm-based `pallet-contracts`. If you're actively building with ink!, it remains a viable path, just with a different maintenance structure than before.

### What Midnight Adds

Neither REVM nor PVM provides privacy. Every contract on Polkadot Hub — whether Solidity-on-REVM or Rust-on-PVM — executes with fully public state and fully public transaction data. For regulated financial applications, enterprise data sharing, healthcare, or compliance-driven workflows, that's a blocker. Midnight's ZK execution model is the answer to use cases that Polkadot Hub cannot serve today.

## 3. Architecture: Polkadot's Stack vs. Midnight

### Polkadot Hub's Contract Architecture

```
Relay Chain (shared security + consensus coordination)
    │
    └─► Polkadot Hub (Asset Hub parachain)
            │
            ├─► REVM: Solidity contracts, full EVM tooling
            │         Hardhat / Foundry / MetaMask / OpenZeppelin
            │
            └─► PVM (pallet-revive): RISC-V native execution
                      Solidity via resolc, or Rust
                      Performance-intensive workloads
```

### Midnight's Architecture

Midnight is not a relay chain and not a parachain framework. It is a single sovereign chain built on the Polkadot SDK — using AURA for block production and GRANDPA for finality, extended with Cardano partner chain components for validator selection. Smart contracts are Compact contracts compiled to ZK circuits. Privacy is structural.

| Concept | Polkadot Hub | Midnight |
|---|---|---|
| Base infrastructure | Relay Chain + Asset Hub parachain | Polkadot SDK — AURA + GRANDPA |
| Primary contract path | Solidity via REVM | Compact contracts (ZK circuits) |
| Secondary contract path | Solidity/Rust via PVM | No equivalent |
| Execution environment | EVM (REVM) or RISC-V VM (PVM), on-chain | ZK circuits run off-chain; proof on-chain |
| Privacy default | Everything public | Everything private unless disclosed |
| Runtime upgrades | Forkless via on-chain governance | Redeploy new contract; migrate state explicitly |
| Cross-chain messaging | XCM (native) | No native equivalent (Q3 2026 roadmap) |
| Gas/fees | Weight-based (ETH-style gas on REVM) | DUST (generated by holding NIGHT) |
| Mainnet status | Fully decentralized | Federated phase; decentralization in progress |

### The Execution Model Difference

On Polkadot Hub's REVM, your Solidity contract runs on every validating node — same as Ethereum. Fully public. On PVM, same model, different VM.

On Midnight, your Compact contract compiles to a ZK circuit. When a user invokes a circuit, it runs locally on their machine. Private data is consumed as witnesses — never on-chain. A ZK proof is submitted. Validators verify the proof only.

| Step | Polkadot Hub (REVM/PVM) | Midnight Compact |
|---|---|---|
| Contract deployed as | EVM bytecode / PVM bytecode, on-chain | ZK circuit definition + ledger state |
| User invokes | Signed transaction | Circuit call via transaction |
| Computation runs | On-chain, every validator re-executes | Locally on user's machine (off-chain) |
| Private inputs | Not possible — all inputs visible | Witness functions — local, never on-chain |
| Chain receives | Signed transaction with calldata | Transaction + ZK proof of correctness |
| Validators do | Re-execute the contract | Verify the proof only (faster) |
| Privacy default | Everything public | Everything private unless `disclose()`d |

> **Proof generation is the new UX-critical latency.** On Polkadot Hub, users wait for block confirmation. On Midnight, they also wait for proof generation — which happens client-side via a **proof server daemon** (a separate process you run alongside your dApp) and currently takes seconds for typical circuits. Optimize for constraint count at compile time, not runtime cost estimation.

## 4. Compact vs. Solidity/ink!: The Language Comparison

### For Solidity Developers (REVM path)

If you're writing Solidity for Polkadot Hub REVM, the Ethereum→Midnight guide in this series covers the Solidity→Compact mapping exhaustively. The short version:

- `function` → `export circuit`
- Storage variables → `export ledger field: Type;`
- `msg.sender` → no equivalent; use ZK-proved key commitment pattern
- `require()` → `assert()`
- Everything public → everything private unless `disclose()`d
- ERC-20 tokens → native UTXO tokens via Zswap
- ETH gas → DUST fees (generated by holding NIGHT)

Refer to that guide for side-by-side code examples.

### For ink!/Rust Developers (PVM path)

If you're coming from ink!, the conceptual distance to Compact is slightly different. Both are non-Solidity languages compiled to a non-EVM target. Both have strong type systems. The differences that matter:

**Compact is TypeScript-like, not Rust-like.** You're leaving the borrow checker behind. Compact has no ownership model, no lifetime annotations, no `Result<T, E>` returns. Error handling is `assert()` — if it fails, the proof fails, the transaction is rejected. The compiler is strict about different things: `disclose()` annotations, circuit bounds, data flow from witnesses.

**The three-part contract structure.** Where ink! has `#[ink(storage)]`, `#[ink(message)]`, and `#[ink(constructor)]`, Compact has three execution contexts:

| Context | What it is | ink! analog |
|---|---|---|
| `export ledger field: Type;` | Public on-chain state | `#[ink(storage)]` struct fields |
| `export circuit fn()` | Provable entry points; run off-chain; proof on-chain | `#[ink(message)]` handlers |
| `witness fn(): Type;` | Private local functions; never on-chain | No equivalent |

**Side-by-side: ink! vs. Compact**

```rust
// ── ink! (Rust) ──────────────────────────────────────
#[ink::contract]
mod vault {
    #[ink(storage)]
    pub struct Vault {
        owner: AccountId,
        value: u64,
    }

    impl Vault {
        #[ink(constructor)]
        pub fn new() -> Self {
            Self { owner: Self::env().caller(), value: 0 }
        }

        #[ink(message)]
        pub fn set_value(&mut self, new_value: u64) {
            assert_eq!(self.env().caller(), self.owner, "Not authorized");
            self.value = new_value;
        }
    }
}
```

```compact
// ── Compact ──────────────────────────────────────────
pragma language_version >= 0.23;
import CompactStandardLibrary;

// On-chain state — flat declarations, NOT a ledger { } block
export ledger authority: Bytes<32>;
export ledger value: Uint<64>;
export ledger round: Counter;    // always include — see §6

// Private key from user's wallet — never on-chain
witness getSecretKey(): Bytes<32>;

// Helper circuit: cryptographic identity commitment
// NEVER use ownPublicKey() for this — it is unconstrained (see §6)
circuit ownerKey(sk: Bytes<32>): Bytes<32> {
    return persistentHash<Vector<3, Bytes<32>>>(
        [pad(32, "vault:owner:"), round as Bytes<32>, sk]
    );
}

constructor(sk: Bytes<32>) {
    authority = disclose(ownerKey(sk));
    round.increment(1);
}

export circuit setValue(newValue: Uint<64>): [] {
    // No Self::env().caller() — prove key knowledge via ZK instead
    assert(ownerKey(getSecretKey()) == authority, "Not authorized");
    value = disclose(newValue);
    round.increment(1);
}
```

| ink! / Polkadot Hub | Compact / Midnight | Notes |
|---|---|---|
| `#[ink::contract] mod` | File IS the contract | One `.compact` file = one contract |
| `#[ink(storage)] struct` | `export ledger field: Type;` | Flat declarations; no block syntax |
| `#[ink(constructor)]` | `constructor()` | Same concept |
| `#[ink(message)]` | `export circuit myFn(): RetType` | `export` = callable in a transaction |

| `Self::env().caller()` | No equivalent | Prove via `persistentHash(witness())` |
| `Self::env().block_timestamp()` | `round: Counter` (ledger field) | No timestamps; use round counter |
| `self.env().emit_event(E{})` | No events | State changes observable via ledger |
| `assert_eq!(a, b, "msg")` | `assert(a == b, "msg")` | Proof fails on violation |
| `ink::Mapping<K, V>` | Stdlib `Map` (bounded) | Bounded — compile-time size |
| `cargo-contract` | `compactc` | The Compact compiler |
| `substrate-contracts-node` / `ink-node` | Midnight DevNet | Local dev chain |
| polkadot.js / PAPI | Midnight.js SDK | TypeScript off-chain SDK |
| `pallet-contracts` / `pallet-revive` | Not applicable | Circuits are the only primitive |
| `chain_extension` | No equivalent | No pallet-level hooks from circuits |
| Weight (`ref_time`, `proof_size`) | Circuit constraint count | Cost is compile-time, not runtime |
| Storage deposit | None | No deposit mechanism |
| Runtime upgrade via governance | Redeploy new contract | No forkless in-place upgrade path |
| XCM cross-chain message | No equivalent (Q3 2026 roadmap) | Not available in current Midnight |

### Key Constraints Compact Adds

**Bounded loops.** EVM/REVM and PVM both meter computation at runtime — you can run unbounded loops and just pay more gas. Compact circuits must have **compile-time-known bounds** on all loops. No dynamic `while (i < array.length)`.

**No cross-contract calls.** Polkadot Hub supports cross-contract calls on both REVM (standard EVM calls) and PVM (via `CallBuilder` in ink!). **Compact 1.0 does not support runtime cross-contract calls.** The `contract` keyword is reserved but unimplemented. Compose logic at compile time via module imports, not at runtime. Design your contracts as monolithic units where Polkadot Hub would be composable.

**Privacy is the default.** Any value derived from a witness function is private. The compiler tracks data flow and requires explicit `disclose()` before private-origin data can reach the public ledger. This is a compile-time guarantee — you cannot accidentally leak private data.

## 5. Privacy Model: What REVM/PVM Can't Do

Everything on Polkadot Hub is public. Every contract storage slot on REVM is readable via RPC. Every PVM contract state is equally visible. Every transaction input and output is on-chain in plaintext.

Midnight inverts this at the protocol level.

### The `disclose()` Annotation

```compact
witness _userBalance(): Uint<64>;   // private: underscore = convention

export ledger total: Uint<64>;
export ledger meetsThreshold: Boolean;

export circuit updateBalance(): [] {
    const bal = _userBalance();

    // ❌ COMPILE ERROR: witness-derived value cannot land on-chain
    total = total + bal;

    // ✅ CORRECT: explicit privacy boundary crossing
    total = total + disclose(bal);

    // ✅ ALSO VALID: prove a property without disclosing the value
    assert(bal >= 1000, "Balance below threshold");
    meetsThreshold = disclose(true);  // only the boolean is on-chain
}
```

`disclose()` is a compiler annotation — not a cast. It signals "I consciously accept that this value is leaving the private domain." The compiler rejects accidental leakage at compile time.

### Three Data Domains vs. Polkadot Hub's One

| Domain | Location | Visible to | Polkadot Hub equivalent |
|---|---|---|---|
| Witness data | User's local machine | Only the user | Not possible |
| Private circuit execution | In-flight during proof generation | Nobody | Not possible |
| Disclosed ledger state | Public ledger | Everyone | All contract storage (the entire model) |

## 6. Tokens, Fees, and the DOT → NIGHT/DUST Shift

### NIGHT and DUST

**NIGHT** is the transferable utility token. Used for staking, governance, and generating DUST. Not spent directly as a fee medium. All NIGHT transfers are publicly visible on-chain — it is deliberately not a privacy coin.

**DUST** is a non-transferable resource generated by holding NIGHT. Used exclusively to pay transaction fees. Expires if unused. Not market-priced — you generate it by holding NIGHT. Fees stay stable regardless of NIGHT's market price.

> **For Polkadot Hub developers:** weight-based fees on REVM can be volatile under congestion — gas prices spike when blocks are full. DUST fees are decoupled from market price entirely. You generate DUST by holding NIGHT; it's a usage allowance, not a fee you pay. For enterprise use cases with predictable cost requirements, this is a meaningful design win.

**DUST cold-start problem.** New wallets have no DUST. Without DUST, they can't pay fees. The solution is **DUST sponsorship**: a dApp holding NIGHT delegates its DUST to cover new users' fees. Design sponsored transactions into your dApp from day one if you expect users who don't hold NIGHT.

| Aspect | Polkadot Hub (REVM) | Midnight |
|---|---|---|
| Fee token | ETH-style gas (weight) | DUST (generated by NIGHT) |
| Fee market | Tip for priority | Dynamic; no competitive mempool |
| Fee predictability | Can spike under congestion | Stable — DUST not traded |
| Failed tx cost | Base fee still charged | Guaranteed-phase DUST forfeited on fallible failure |
| Storage cost | No deposit (REVM) | No deposit mechanism |

### Custom Tokens

Midnight supports two token models:

**UTXO-based tokens (via Zswap):** Tokens live as discrete unspent outputs. Shielding is native. Atomic multi-asset swaps are protocol-level. The primary model for private transfers.

**Account-style tokens (via Compact contract storage):** Ledger fields track balances. Useful for governance tokens with delegation, complex DeFi state machines.

## 7. Transactions and the Two-Phase Execution Model

Polkadot Hub REVM transactions are atomic: fully succeed or fully revert (minus base fee). Midnight splits execution into three stages with importantly different semantics — **this is the most critical operational difference to understand.**

**Stage 1 — Well-formedness check:** Runs against no state. Verifies all ZK proofs and that offers are balanced. A bad proof or unbalanced offer → rejected outright, not included. No DUST spent.

**Stage 2 — Guaranteed phase:** Runs against ledger state. **DUST fees are collected here.** If this phase fails, the transaction is *not included at all*.

**Stage 3 — Fallible phase:** Also runs against ledger state. If this fails, the guaranteed phase effects **still apply**, and the transaction is recorded as a **partial success**. Fees are forfeited.

**Why this matters:** Unlike REVM's clean revert, a Midnight transaction can land in a partially-applied state. Design circuits so partial success is always a safe state to be in. Put must-succeed logic in the guaranteed phase; put speculative effects in the fallible phase — and never put a step there that a guaranteed-phase commitment already assumed would complete.

## 8. Appchain vs. Smart Contract: The Polkadot Decision Doesn't Exist Here

One of the most distinctive aspects of Polkadot development is the architectural choice: deploy a smart contract on an existing parachain (Polkadot Hub), or build a dedicated appchain with full parachain sovereignty? Appchains give you custom modules, custom fee tokens, custom governance, full control.

**Midnight eliminates this choice.** Compact contracts are the only building block. There are no custom modules, no runtime to author, no consensus to configure. If you need appchain-level control, you'd need a separate chain. Midnight's value proposition is privacy, not sovereignty.

| Polkadot Appchain Feature | Available in Midnight? | Notes |
|---|---|---|
| Custom fee token | No — fees are DUST | NIGHT/DUST is the only fee model |
| Custom SDK modules | No — circuits are the primitive | No module layer |
| Custom governance | No — Midnight has its own | Can build governance logic in circuits |
| Custom consensus | No | AURA + GRANDPA; not configurable |
| Upgrade via governance | No — redeploy new contract | No forkless in-place upgrade |
| XCM integration | No — roadmap item | Not available in current Midnight |

**When to use Midnight vs. a Polkadot appchain:**

Use Midnight when privacy is the core requirement — sensitive data, regulated compliance, enterprise confidentiality, selective disclosure — and the logic is self-contained without cross-chain IBC/XCM flows.

Use a Polkadot appchain when you need custom fee tokens, custom modules, XCM cross-chain composability, or full control over your chain's runtime.

## 9. Security: What Changes, What Disappears, What's New

### Polkadot Hub Vulnerabilities That Don't Apply on Midnight

- **Reentrancy via cross-contract calls:** Compact circuits are closed computations with no external calls mid-execution. No reentrancy vector exists in Compact 1.0.
- **Missing signer checks (REVM):** Replaced structurally — there is no `msg.sender` to forget checking. If you omit the `assert`, the circuit simply has no access control.
- **Storage deposit accounting errors:** No storage deposit mechanism.
- **Front-running via mempool:** Shielded transactions don't expose amounts or intent pre-broadcast.

### Security Patterns That Still Apply

- **`assert()` for invariants:** Same concept as `require()` on REVM or `assert_eq!()` in ink!. Proof fails on violation.
- **Access control completeness:** Every restricted circuit needs its own `assert(ownerKey(getSecretKey()) == authority)`. No global modifier or inherited context.
- **Round counter increment:** Always include `export ledger round: Counter;` and call `round.increment(1)` in every circuit that modifies state. This provides replay protection and transaction unlinkability.

### New Attack Surfaces

**`ownPublicKey()` and unconstrained public key comparisons** are the most dangerous footgun for Polkadot Hub developers moving to Midnight. In a Compact circuit, calling a stdlib function to retrieve the caller's public key compiles to an **unconstrained private input** — a value the prover can set freely, including to the stored authority commitment, without knowing any secret. An attacker sets their "public key" to the stored value, the comparison passes, and access is granted. Always use the `persistentHash` commitment scheme with domain separation as shown in §4.

**Disclosure mistakes:** Using `disclose()` on data you didn't intend to make public. The compiler catches accidental leakage of witness data but can't catch *intentional* disclosure you didn't think through. Audit every `disclose()` call.

**Commitment brute-forcing:** Small-domain commitments (booleans, small integers) without a random salt are trivially invertible. Use `persistentCommit()` with randomness for small-domain secrets.

**Witness implementation bugs:** `compactc` validates the circuit interface and data flow. It cannot validate your TypeScript witness implementations. The `@openzeppelin/compact-contracts` library ships test witnesses only — not production witnesses. If you use it, author and audit your own.

**Partial-success states:** Given the two-phase execution model (§7), a fallible-phase failure can leave your contract in a partially-applied state with no clean revert. Design circuit boundaries so partial success is always safe.

### Project Structure

```
my-midnight-dapp/
├── contracts/
│   ├── my_contract.compact          # Compact source
│   └── managed/                     # Compiler output — do not edit
│       └── my_contract/
│           ├── contract/index.cjs   # Generated TypeScript stubs (your typed ABI)
│           ├── keys/                # ZK proving/verifying keys
│           └── zkir/                # ZK intermediate representation
└── src/
    ├── index.ts                     # dApp logic using Midnight.js SDK
    └── witnesses.ts                 # Implement witness functions here
```

### Development Workflow

1. **Write contract:** Author `.compact` file. Declare ledger fields, witness function signatures, exported circuits.
2. **Compile:** Run `compactc`. Errors on `disclose()` violations, non-constant loop bounds, and type mismatches.
3. **Start proof server:** Run the proof server daemon. Your SDK calls route through it for proof generation. Budget for this process in deployment architecture.
4. **Implement witnesses:** Compiler generates TypeScript interfaces; you implement the bodies.
5. **Test:** Run against local Midnight DevNet.
6. **Deploy:** Midnight.js SDK. Deployer's wallet pays DUST. Contract is immutable post-deployment.

| Runtime upgrade via governance | Redeploy new contract | No forkless in-place upgrade |
| XCM cross-chain message | No equivalent (Q3 2026 roadmap) | Not available in current Midnight |

## 12. Suggested Learning Path

1. **Install the toolchain** → start the proof server → clone `example-hello-world` → deploy to local DevNet. Get the compile → proving-keys → TypeScript-driver loop in your fingers. ([docs.midnight.network/getting-started](https://docs.midnight.network/getting-started))
2. **Work the bulletin-board tutorial** ([docs.midnight.network/tutorials/bboard](https://docs.midnight.network/tutorials/bboard)). Study the `publicKey` helper circuit carefully — it's the canonical correct access control pattern. Understand why the domain tag and round counter are in the commitment.
3. **Read the smart contract security page** ([docs.midnight.network/compact/smart-contract-security](https://docs.midnight.network/compact/smart-contract-security)). Specifically: the `ownPublicKey()` warning and commitment brute-forcing.
4. **Read the Compact language reference and standard library** ([docs.midnight.network/compact](https://docs.midnight.network/compact)). Pay attention to: `Counter`, `Map`, `MerkleTree`, `Maybe`.
5. **Read the "How Midnight Works" concept pages** — transaction semantics (especially guaranteed/fallible phases), Zswap, DUST architecture, Kachina.
6. **Build something with a native token** to feel the Zswap UTXO model, then add selective disclosure to feel the compliance angle.
7. **Explore `@openzeppelin/compact-contracts`** for `Ownable`, `Pausable`, and `FungibleToken` patterns — actively maintained but flagged experimental. Author your own witnesses; do not reuse the test witnesses.
