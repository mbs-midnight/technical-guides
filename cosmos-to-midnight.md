# From Cosmos to Midnight — Developer Field Guide

*You already think in actor-model messages and IBC. Now add zero-knowledge proofs and privacy by default.*

**Midnight Foundation | June 2026**

> **Version note:** Midnight mainnet launched March 31, 2026 in a federated phase — a curated set of trusted node operators (IOG, Google Cloud, Vodafone) runs the network while decentralization via Cardano SPO onboarding is completed. Compact targets `language_version >= 0.23`. Cross-contract calls are reserved in the language spec but not yet implemented in Compact 1.0. Always verify against [docs.midnight.network](https://docs.midnight.network) before shipping.

---

## 1. Why Midnight? The Pitch for Cosmos Developers

The Cosmos thesis is sovereignty: every application should own its chain, its consensus, its fee token, and its upgrade path. You've built around that — whether writing CosmWasm contracts, authoring Cosmos SDK modules, or designing IBC-enabled interchain applications. The appchain model gives you power that monolithic chains never could.

What the Cosmos appchain model doesn't give you natively is privacy at the smart contract layer. Every CosmWasm contract's storage is public. Every IBC packet is visible. Every `info.sender` is on-chain. For regulated financial applications, enterprise data sharing, confidential business logic, or healthcare use cases, that public-by-default model is a compliance blocker.

**One-line pitch:** Midnight is built exactly for those cases — programmable on-chain logic where the underlying data must remain private. It uses zero-knowledge proofs as the execution model, not as an add-on. Privacy is structural, not policy-based. And like Cosmos, Midnight runs on the Polkadot SDK (formerly Substrate) — so the node infrastructure is more familiar than you might expect.

### What Carries Over from Cosmos

- **Rust mental model.** Compact is TypeScript-like, but your type-safety instincts about explicit errors and compile-time correctness transfer well.
- **Actor model intuitions.** CosmWasm's message-passing, no-reentrancy design maps conceptually to Compact's circuit model: logic processes inputs and produces results rather than imperatively calling back into external contracts.
- **Separation of concerns.** Cosmos separates SDK modules from contract logic; Midnight separates circuits (on-chain-provable) from witnesses (private, off-chain). Different mechanism, same instinct.
- **Substrate under the hood.** Midnight is built on the Polkadot SDK using AURA block production and GRANDPA finality. Node and runtime concepts are familiar.
- **TypeScript off-chain tooling.** Midnight.js SDK is TypeScript, comfortable for anyone who has used CosmJS or TS-based Cosmos clients.

### What Needs Rewiring

- **CosmWasm / Rust → Compact:** new language, different compiler target (ZK circuits, not Wasm)
- **`info.sender` / `MessageInfo` → does not exist:** access control is proved via ZK, not transaction metadata
- **Entry points:** `instantiate` / `execute` / `query` / `migrate` → `constructor` / `export circuit` / ledger state read directly / redeploy
- **IBC → no native equivalent:** cross-chain is a roadmap item, not a live feature
- **ATOM / chain-native fees → DUST fees:** non-transferable resource generated by holding NIGHT
- **`cw-storage-plus` → `export ledger` fields:** Compact manages persistence with typed fields, no `.save()` / `.load()`
- **On-chain Wasm execution → off-chain ZK proof generation:** validators verify proofs, not re-execute
- **`migrate` entry point → redeploy:** no in-place contract migration

## 2. Architecture: Cosmos Stack vs. Midnight

### The Cosmos Stack

- **CometBFT / Tendermint BFT:** Consensus engine. Instant finality, fast block times, Proof-of-Stake validation. Not where application logic lives.
- **Cosmos SDK:** Application framework. Modules (`x/bank`, `x/staking`, `x/gov`) compose into an appchain. The chain's runtime is a composition of SDK modules.
- **x/wasm (CosmWasm):** A Cosmos SDK module that enables Wasm smart contracts. Contracts run inside it with access to other SDK modules via message dispatching.
- **IBC:** Inter-Blockchain Communication protocol. IBC Eureka (v2, launched April 2025) connects 115+ chains including Ethereum via ZK light client proofs, with Solana and L2 integrations in progress.

### Midnight's Architecture

Midnight is a purpose-built privacy blockchain — not a Cosmos SDK chain, not an appchain framework. It's a single sovereign chain built on the Polkadot SDK (Substrate), using AURA for block production and GRANDPA for finality, extended with Cardano partner chain components. Cardano SPOs can participate as Midnight block validators — Midnight is the first Cardano partner chain.

Where Cosmos decomposes a chain into composable SDK modules, Midnight's architecture separates public on-chain state from private local state at the protocol level. Privacy is structural — not a module you add on top. Compact contracts compile to ZK circuits rather than Wasm bytecode.

| Concept | Cosmos / CosmWasm | Midnight |
|---|---|---|
| Consensus | CometBFT / Tendermint BFT | AURA + GRANDPA (Polkadot SDK) |
| Application layer | Cosmos SDK modules | No module layer; circuits are the primitive |
| Smart contracts | CosmWasm (x/wasm), Wasm/Rust | Compact contracts, ZK circuits |
| Contract execution | On-chain Wasm VM, all validators re-execute | Off-chain proof generation; validators verify proof |
| Privacy default | Everything public | Everything private unless disclosed |
| Cross-chain | IBC (115+ chains, IBC Eureka to Ethereum) | No native equivalent (roadmap) |
| Upgrade path | `migrate` entry point + SDK governance | Redeploy; no in-place migration |
| Fee token | ATOM or chain-native token | DUST (generated by holding NIGHT) |
| Token model | Native denoms + CW-20 contracts | UTXO-primary (Zswap) + account-style Compact tokens |
| Mainnet status | Fully decentralized | Federated phase (trusted operators); decentralization in progress |

### The Execution Model Inversion

In CosmWasm, when a user submits an execute message, the Wasm VM runs your contract on every validating node. Re-execution is how consensus is reached. Everything is visible because it must be.

In Midnight, the contract logic runs locally on the user's machine. Private inputs (spending keys, off-chain data) are consumed as witnesses. A ZK proof is generated — proving the computation was done correctly without revealing the private inputs. That proof, not the computation, goes to the chain. Validators verify the proof. Private state never appears on-chain.

| Step | CosmWasm | Midnight Compact |
|---|---|---|
| User submits | Signed transaction with execute msg | Transaction with circuit call + ZK proof |
| Computation runs | On every validator (on-chain Wasm VM) | Locally on user's machine (off-chain) |
| Private inputs | Not possible — all inputs visible | Witness functions — local, never on-chain |
| Chain validates | Re-executes the Wasm contract | Verifies the ZK proof (fast, no re-execution) |
| State update | Contract storage mutated in-place | Ledger state updated; UTXOs created |

> **Proof generation is the new UX-critical latency.** CosmWasm users wait for block inclusion (~2–6 seconds on most chains). Midnight users also wait for proof generation — which happens client-side via a **proof server daemon** (a separate process you run alongside your dApp) and currently takes seconds for typical circuits. Unlike gas limits, you optimize for constraint count at compile time.

## 3. CosmWasm vs. Compact: The Language Comparison

CosmWasm contracts are Rust crates that compile to WebAssembly and expose standardized entry points. Compact is a TypeScript-like DSL that compiles to ZK arithmetic circuits. The languages share a strong-typing philosophy and a preference for explicit error handling, but the runtime targets and execution models are entirely different.

### CosmWasm Contract Structure

```rust
use cosmwasm_std::{entry_point, DepsMut, Deps, Env, MessageInfo,
                   Response, StdResult, Binary, to_json_binary};
use cw_storage_plus::Item;

const OWNER: Item<String> = Item::new("owner");
const VALUE: Item<u64>    = Item::new("value");

#[entry_point]
pub fn instantiate(deps: DepsMut, _env: Env, info: MessageInfo,
                   _msg: InstantiateMsg) -> StdResult<Response> {
    OWNER.save(deps.storage, &info.sender.to_string())?;
    VALUE.save(deps.storage, &0u64)?;
    Ok(Response::new())
}

#[entry_point]
pub fn execute(deps: DepsMut, _env: Env, info: MessageInfo,
               msg: ExecuteMsg) -> StdResult<Response> {
    match msg {
        ExecuteMsg::SetValue { new_value } => {
            let owner = OWNER.load(deps.storage)?;
            if info.sender.to_string() != owner {
                return Err(StdError::generic_err("Not authorized"));
            }
            VALUE.save(deps.storage, &new_value)?;
            Ok(Response::new().add_attribute("action", "set_value"))
        }
    }
}

#[entry_point]
pub fn query(deps: Deps, _env: Env, msg: QueryMsg) -> StdResult<Binary> {
    match msg {
        QueryMsg::GetValue {} => to_json_binary(&VALUE.load(deps.storage)?),
    }
}
```

### Compact Equivalent

```compact
pragma language_version >= 0.23;
import CompactStandardLibrary;

// On-chain state — each field declared individually (NOT a ledger { } block)
export ledger authority: Bytes<32>;   // stored identity commitment
export ledger value: Uint<64>;
export ledger round: Counter;          // anonymity counter — always include

// Witness: caller's private key — never leaves their machine
witness secretKey(): Bytes<32>;

// Helper circuit: derives a tamper-proof identity commitment
// Uses domain separation and the round counter for unlinkability
circuit ownerKey(sk: Bytes<32>): Bytes<32> {
    return persistentHash<Vector<3, Bytes<32>>>(
        [pad(32, "myapp:owner:pk:"), round as Bytes<32>, sk]
    );
}

constructor(deployerSk: Bytes<32>) {
    authority = disclose(ownerKey(deployerSk));
    round.increment(1);
}

// Entry point: replaces execute handler
export circuit setValue(newValue: Uint<64>): [] {
    // No info.sender — prove key ownership via ZK instead
    assert(ownerKey(secretKey()) == authority, "Not authorized");
    value = disclose(newValue);
    round.increment(1);
}

// No query entry point — read ledger state directly via SDK off-chain:
// const state = await contract.ledger();  // state.value, state.authority, etc.
```

**Ledger syntax note:** Each ledger field uses its own `export ledger fieldName: Type;` statement. There is no `ledger { }` block syntax in current Compact — that appeared in pre-mainnet versions and will fail to compile against the current toolchain.

**`round.increment(1)` note:** `round` is declared as type `Counter` (a stdlib ledger type with an `increment()` method). Using `round = round + 1` is a type error. Always use `round.increment(1)`.

### Entry Point Mapping

| CosmWasm Entry Point | Compact Equivalent | Key Difference |
|---|---|---|
| `instantiate(deps, env, info, msg)` | `constructor()` | No `info.sender`; no `deps`; sets ledger fields directly |
| `execute(deps, env, info, msg)` | `export circuit myFn(params)` | No `info.sender`; access via ZK key proof |
| `query(deps, env, msg)` | No entry point — read ledger state off-chain | Read via SDK: `contract.ledger()` — no on-chain query circuit needed |
| `migrate(deps, env, msg)` | No equivalent | Redeploy a new contract; migrate state off-chain |
| `sudo(env, msg)` | No equivalent | No privileged chain-module entry point |
| `reply(deps, env, msg)` | No equivalent | No submessage callback model |
| `ibc_*` entry points | No equivalent (roadmap) | IBC integration not yet available |

### Actor Model vs. ZK Circuit Model

CosmWasm's actor model is one of its most thoughtful design decisions: contracts communicate by returning `CosmosMsg` objects rather than calling other contracts directly. This prevents reentrancy by design — there are no nested calls mid-execution, only depth-first message dispatch after each contract returns.

Compact circuits are even more constrained: they are **closed computations with no external calls at all**. A circuit cannot dispatch messages to other contracts, query other contracts, or interact with SDK modules at runtime. It processes its inputs (ledger state + circuit parameters + witness outputs) and produces updated ledger state and a ZK proof. That's the full extent of what it can do.

**Cross-contract calls at runtime are not implemented in Compact 1.0.** The `contract` keyword is reserved in the language spec but non-functional. What was a multi-contract composition in CosmWasm becomes a single compiled Compact contract with imported modules. Design accordingly: what CosmWasm would do at message dispatch time, Compact does at compile time.

| Aspect | CosmWasm Actor Model | Compact Circuit Model |
|---|---|---|
| External calls | Return `CosmosMsg` for depth-first dispatch | No external calls — circuits are closed |
| Submessages | `reply` callbacks | No equivalent |
| Cross-contract query | `QueryRequest` via `deps.querier` | No equivalent — read off-chain via SDK |
| Cross-contract call | `CosmosMsg::Wasm(WasmMsg::Execute{...})` | **Not implemented in Compact 1.0** |
| Reentrancy | Prevented by actor model (deferred dispatch) | Prevented structurally (no calls possible) |
| SDK module access | `CosmosMsg::Bank`, `Staking`, `Gov`, etc. | No direct module access; use Zswap for tokens |
| Event emission | `.add_attribute()` / `.add_event()` | No events; observe ledger state changes |

## 4. State and Storage: From cw-storage-plus to Ledger Fields

CosmWasm's `cw-storage-plus` is one of the most ergonomic storage libraries in the Rust blockchain ecosystem: `Item<T>`, `Map<K, V>`, `IndexedMap`, `SnapshotMap` — composable, type-safe primitives that serialize to a key-value store. Everything is public by default; all storage is readable via RPC.

Midnight's state model is structurally different. Contract state is declared with `export ledger` fields — typed, directly updated by circuits, with no storage abstractions to import and no `.save()` / `.load()` calls. And privacy is enforced by the compiler: any value derived from a witness function requires explicit `disclose()` before it can reach ledger state.

### Storage Mapping

| CosmWasm (cw-storage-plus) | Compact | Notes |
|---|---|---|
| `Item<T>` | `export ledger field: Type;` | Single typed value |
| `Map<K, V>` | Stdlib `Map` type (bounded) | Bounded; use Merkle patterns for large sets |
| `IndexedMap` | No equivalent | Indexing is off-chain; query via SDK |
| `SnapshotMap` | No equivalent | History via external indexers |
| `item.save(deps.storage, &v)?` | `ledger.field = disclose(v)` | `disclose()` required for private-origin values |
| `item.load(deps.storage)?` | `ledger.field` | Direct access in circuit body |
| `item.may_load(...)?` | Typed fields cannot be absent | Compact ledger fields are always initialized |
| `item.remove(deps.storage)` | No equivalent | Ledger fields are not deletable |

### The `disclose()` Annotation: Privacy by Compiler

CosmWasm has no concept of private inputs — every argument, every storage value, every SDK query result is fully public. Compact enforces a privacy boundary at the type-system level. The compiler tracks the origin of every value; anything produced by or derived from a witness function is private. It cannot reach the ledger without explicit annotation.

```compact
witness _userScore(): Uint<64>;   // private origin

export circuit submitScore(): [] {
    const score = _userScore();

    // ❌ COMPILE ERROR: witness-derived value cannot land on-chain directly
    ledger.lastScore = score;

    // ✅ CORRECT: explicit privacy boundary crossing
    ledger.lastScore = disclose(score);

    // ✅ ALSO VALID: prove a property about score without disclosing the value
    assert(score >= 500, "Score below minimum threshold");
    ledger.qualifies = disclose(true);  // only the boolean hits the chain
}
```

That third pattern — proving a property about private data without revealing the underlying value — is the core use case Midnight is designed for. The score stays private; only the fact that it meets the threshold becomes public. This is what makes regulatory compliance without full disclosure possible.

## 5. Access Control: From `info.sender` to ZK-Proved Keys

`info.sender` is the cornerstone of CosmWasm access control. It's available in every `execute`, `instantiate`, and `migrate` call. Validated by the chain runtime — you don't need to verify it yourself.

Midnight has no `info.sender`. It doesn't exist because exposing it would link identities to contract interactions, undermining the privacy model. Instead, access control is proved via knowledge of a private key.

### A Critical Security Warning: Don't Use `ownPublicKey()` for Access Control

Before showing the correct pattern, a hard warning — because the wrong pattern is both intuitive and broken.

A naive first attempt looks like: call a stdlib function to retrieve the user's public key inside the circuit, compare it against a stored value. This is wrong in a critical way. In a Compact circuit, a function like `ownPublicKey()` compiles to an **unconstrained private input** — a value the prover sets freely, with zero cryptographic obligation to prove they own the corresponding private key. An attacker can supply any public key, claim ownership, and pass the check. The ZK proof proves nothing useful.

The correct pattern requires:

1. **A witness function** that retrieves the user's actual secret key locally (off-chain, never on-chain).
2. **A helper circuit** (not a stdlib function call) that derives a commitment from that secret key using `persistentHash` with domain separation.
3. **The `assert`** compares the derived commitment against what's stored in the ledger.

The ZK proof then cryptographically demonstrates: "I know the secret key that produces this commitment" — without revealing the key.

### Pattern Comparison

```rust
// CosmWasm: info.sender access control
#[entry_point]
pub fn execute(deps: DepsMut, _env: Env, info: MessageInfo,
               msg: ExecuteMsg) -> StdResult<Response> {
    let owner = OWNER.load(deps.storage)?;
    if info.sender.to_string() != owner {
        return Err(StdError::generic_err("Not authorized"));
    }
    // ... restricted logic
}
```

```compact
// Compact: ZK-proved key ownership (correct pattern)
export ledger authority: Bytes<32>;   // stored commitment — NOT a raw public key
export ledger round: Counter;

witness secretKey(): Bytes<32>;

// Helper circuit: builds an identity commitment with domain separation
// The round counter ensures each commitment is unique to this contract instance,
// preventing key-reuse attacks across contracts
circuit ownerKey(sk: Bytes<32>): Bytes<32> {
    return persistentHash<Vector<3, Bytes<32>>>(
        [pad(32, "myapp:owner:pk:"), round as Bytes<32>, sk]
    );
}

export circuit restrictedAction(param: Uint<64>): [] {
    // Prove: "I know the sk that produced authority, for this round"
    // The ZK proof verifies this without the chain ever seeing sk
    assert(ownerKey(secretKey()) == authority, "Not authorized");
    value = disclose(param);
    round.increment(1);
}
```

> **Every restricted circuit needs its own explicit `assert`.** There is no inherited authorization context between circuits — no `cw-ownable` decorator, no shared modifier. If you add a new exported circuit and forget the `assert`, it is callable by anyone. Make the access check the first line of every restricted circuit.

### Multi-Role Access Control

The CosmWasm pattern of `Map<Addr, Role>` becomes multiple authority fields in ledger state, each with its own commitment and its own checking circuit:

```compact
export ledger admin: Bytes<32>;
export ledger operator: Bytes<32>;
export ledger round: Counter;

witness secretKey(): Bytes<32>;

circuit roleKey(domainTag: Bytes<32>, sk: Bytes<32>): Bytes<32> {
    return persistentHash<Vector<3, Bytes<32>>>(
        [domainTag, round as Bytes<32>, sk]
    );
}

export circuit adminAction(param: Uint<64>): [] {
    assert(roleKey(pad(32, "myapp:admin:"), secretKey()) == admin, "Not admin");
    // ... admin logic
    round.increment(1);
}

export circuit operatorAction(param: Uint<64>): [] {
    assert(roleKey(pad(32, "myapp:operator:"), secretKey()) == operator, "Not operator");
    // ... operator logic
    round.increment(1);
}
```

Note the domain tag differs per role — this prevents a single secret key from being valid across both roles if an attacker knows one commitment and tries to derive another.

## 6. Tokens, Fees, and the ATOM → NIGHT/DUST Shift

### NIGHT and DUST: The Midnight Protocol Tokens

**NIGHT** is the transferable utility token. Used for staking (network security via Cardano SPO participation), governance, and passive DUST generation. Closest analogy to ATOM — but unlike ATOM, NIGHT is not spent directly as a fee medium. All NIGHT transfers are publicly visible on-chain. It is deliberately not a privacy coin.

**DUST** is a non-transferable resource generated by holding NIGHT. Used exclusively to pay transaction fees. Expires if unused. Not market-priced — generated passively as a function of NIGHT holdings. Transaction costs stay stable regardless of NIGHT's market price.

> **For Cosmos developers:** IBC fee volatility and gas pricing across different chains is a real operational headache. DUST sidesteps this entirely — you generate it by holding NIGHT, it's not traded, and transaction costs don't spike during congestion the way ATOM-denominated fees can. For enterprise use cases with predictable cost requirements, this is a meaningful design win.

**Key DUST properties:**
- **Shielded.** Fee payment doesn't expose your wallet address.
- **Non-transferable.** Cannot be sent between wallets — but can be *delegated* to sponsor others' transactions.
- **Decaying.** Unused DUST bleeds off; move your NIGHT and it decays to zero.
- **Renewable.** Consume it and it refills over time based on NIGHT holdings.

**The cold-start problem.** New wallets have no DUST. Without DUST, they cannot pay fees. The solution is **DUST sponsorship**: a dApp holding NIGHT delegates its DUST to cover new users' fees. The Midnight SDK exposes this as a two-phase transaction balancing model — the user covers token flows, the sponsor covers DUST. Design sponsored transactions into your dApp from day one if you expect users who don't hold NIGHT.

### Token Model Comparison

| Property | Cosmos (native denom + CW-20) | Midnight UTXO tokens (Zswap) | Midnight account-style tokens |
|---|---|---|---|
| Native token transfer | `CosmosMsg::Bank::Send` | Zswap UTXO: consume input, create output | Circuit mutates ledger state |
| Fungible token standard | CW-20 (contract-based) | 256-bit token type derived from contract address | Defined by the Compact contract |
| Balance storage | `x/bank` module or CW-20 mapping | UTXOs (shielded or unshielded) | `export ledger` field |
| Privacy option | None — all amounts public | Shielded UTXOs hide value and owner | Public by default; `disclose()` controls |
| Atomic multi-token swap | Multiple messages / DEX contract | Native via Zswap protocol | Circuit-level composition |
| IBC token bridging | ICS-20 / cw20-ics20 | No native equivalent (roadmap) | No native equivalent (roadmap) |
| Approve / allowance | CW-20 `increase_allowance` | No approve — ZK proof owns the spend | No approve — ZK proof owns the spend |
| Best for | General-purpose token use cases | Private transfers, privacy-first DeFi | Complex DeFi, governance, gaming |

### Zswap: Atomic Multi-Asset Swaps

Zswap is Midnight's native atomic swap protocol, built into the transaction layer — not a contract or module on top. The core concept is an **offer**: a bundle specifying input UTXOs consumed and output UTXOs created, with a balance vector tracking net value flow per token type. Two parties' offers merge non-interactively. The matcher verifies the combined balance nets to zero without seeing the amounts.

No `approve()` step. No front-running exposure. Token identity is cryptographically derived from the issuing contract's address — no global registry needed.

> **Zswap implementation note:** The official docs flag this layer as not yet performance-optimized and subject to further revision. The conceptual model is stable; verify specific API surface before building production integrations.

### IBC: The Biggest Functional Gap

IBC is Cosmos's superpower. IBC Eureka (v2, launched April 2025) uses ZK light client proofs to connect Ethereum to all Cosmos chains, with Solana and L2 integrations in development. Cross-chain contract calls via interchain accounts, ICS-20 token flows, and complex multi-hop routing are live production features.

Midnight currently has **no equivalent of IBC**. There are no `ibc_*` entry points in Compact, no ICS-20 token bridge, no interchain account protocol. Midnight cross-chain interoperability is a roadmap item — LayerZero integration is targeted for the Q3 2026 "Hua" phase, which would let Midnight function as privacy infrastructure for any chain connecting through LayerZero. That's not live today.

If your application depends on IBC for cross-chain token flows, interchain accounts, or cross-chain contract calls, that functionality does not port to current Midnight. Applications that are self-contained — where privacy is the primary requirement and cross-chain movement is secondary — are the best fit for Midnight today.

## 7. Common CosmWasm Patterns in Compact

### Ownership Transfer

```rust
// CosmWasm: two-step pending ownership transfer (cw-ownable)
// PENDING_OWNER stored, then claimed by new owner signing a tx
```

```compact
// Compact: direct transfer by current owner, ZK proof IS the authorization
export circuit transferOwnership(newOwnerSk: Bytes<32>): [] {
    // Current owner authorizes the transfer
    assert(ownerKey(secretKey()) == authority, "Not authorized");
    // New owner's commitment replaces the old one; takes effect immediately
    // No pending-owner pattern needed — the ZK proof verifies intent
    authority = disclose(ownerKey(newOwnerSk));
    round.increment(1);
}
```

### Time-Locked Logic

CosmWasm's `env.block.time` gives you block timestamps for vesting, auctions, and time-locked logic. Midnight circuits don't expose block timestamps — doing so would leak timing information about when transactions occurred, compromising privacy.

```rust
// CosmWasm: time-locked vesting
if env.block.time < vesting_end_time {
    return Err(StdError::generic_err("Not yet vested"));
}
```

```compact
// Compact: round counter for relative ordering
export ledger vestingEndRound: Uint<64>;
export ledger round: Counter;

export circuit claimVested(): [] {
    assert(ownerKey(secretKey()) == authority, "Not authorized");
    assert(round as Uint<64> >= vestingEndRound, "Not yet vested");
    // ... claim logic
    round.increment(1);
}
// Rounds give relative ordering.
// For absolute wall-clock time, inject timestamps via a witness function
// from off-chain, then validate with an oracle commitment pattern.
```

### Submessage / Sub-contract Instantiation

CosmWasm's submessage pattern — instantiate a new contract, get its address back via `reply` — enables factory patterns and multi-contract deployments. This has no runtime equivalent in Compact because circuits cannot dispatch messages during execution.

The alternative: deploy contracts independently using the Midnight.js SDK off-chain, then store relevant contract addresses or authority commitments in each contract's ledger state as configuration. Cross-contract interaction at the Compact level is achieved by importing modules at compile time.

### CW-20 Token Interactions

CosmWasm DeFi contracts typically interact with CW-20 tokens via message dispatch: user approves an allowance, contract calls `TransferFrom`. This two-step dance is a common source of bugs.

Midnight token interactions use the Zswap UTXO model. Token movements are expressed as UTXO inputs and outputs in the transaction — not as cross-contract message dispatch. There's no `approve` step because UTXO ownership is proved via ZK proof, not via a pre-authorized allowance. The entire approve/allowance attack surface is eliminated.

## 8. Appchain vs. Smart Contract: The Cosmos Decision Doesn't Exist Here

One of the most distinctive aspects of Cosmos development is the deliberate architectural choice: smart contract on an existing chain, or its own sovereign appchain? Appchains give you custom modules, custom fee tokens, custom governance, and full control. Contracts are faster to deploy but less powerful.

**Midnight eliminates this choice.** Compact contracts are the only building block. There are no custom modules, no runtime to author, no consensus to configure. If you need appchain-level control (custom fee models, custom consensus, full governance), you'd need a separate chain. Midnight's value proposition is privacy, not sovereignty.

| Cosmos Appchain Feature | Available in Midnight? | Notes |
|---|---|---|
| Custom fee token | No — fees are DUST | NIGHT/DUST is the only fee model |
| Custom SDK modules | No — circuits are the primitive | No module layer available |
| Custom governance | No — Midnight has its own | Can build governance logic in circuits |
| Custom consensus | No | AURA + GRANDPA; not configurable |
| Custom state transition function | No | Compact contracts define logic, not runtime |
| Upgrade via governance vote | No — redeploy new contract | No forkless in-place upgrade path |
| IBC integration | No — roadmap item | Not available in current Midnight |

**When to use Midnight vs. a Cosmos appchain:**

Use Midnight when privacy is the core requirement — sensitive data, regulated compliance, enterprise confidentiality, or selective disclosure — and the logic is self-contained without cross-chain IBC flows.

Use a Cosmos appchain when you need custom fee tokens, custom modules, IBC cross-chain composability, or full control over your chain's runtime.

Consider both when you need IBC composability AND privacy — a future Midnight IBC integration would serve this, but today it's not available. An interim architecture might use a Cosmos appchain for public state and IBC, with Midnight for the privacy-sensitive subset.

## 9. Transactions and the Two-Phase Execution Model

This is absent from most Midnight documentation and from the CosmWasm mental model — read carefully.

### What a Midnight Transaction Contains

- One or more circuit invocations with their ZK proofs
- UTXO inputs being consumed (shielded or unshielded)
- UTXO outputs being created (payments, change, new token UTXOs)
- Nullifiers for spent UTXOs (prevent double-spend)
- DUST fee payment

### The Two-Phase Execution Model

CosmWasm extrinsics are all-or-nothing: they fully succeed or the entire transaction reverts (though gas is still charged). Midnight splits execution into three stages with importantly different semantics:

**Stage 1 — Well-formedness check:** Runs against no state. Verifies all ZK proofs and that offers are balanced (accounting for fees and mints). A bad proof or unbalanced offer → rejected outright, not included.

**Stage 2 — Guaranteed phase:** Runs against ledger state. **DUST fees are collected here.** If this phase fails, the transaction is *not included at all*.

**Stage 3 — Fallible phase:** Also runs against ledger state. If this fails, the effects of the guaranteed phase **still apply**, and the transaction is recorded as a **partial success**. Fees are forfeited.

**Why this matters:** Unlike CosmWasm's clean revert, a Midnight transaction can land in a partially-applied state. Design your circuits so that partial success is always a safe state to be in. Put DUST payment and must-succeed invariants in the guaranteed phase. Put speculative or best-effort effects in the fallible phase — and never put a step there that a guaranteed-phase commitment already assumed would complete.

## 10. Toolchain and Development Environment

Cosmos developers work in a Rust-heavy environment: `cargo` as the build system, `cosmwasm-check` for contract validation, CosmJS or PAPI for off-chain TypeScript interactions, `wasmd` or chain-specific nodes for local testing. Midnight's toolchain is TypeScript-centric at the off-chain layer with a dedicated Compact compiler for on-chain logic — and a proof server daemon with no Cosmos equivalent.

### Tool Mapping: Cosmos → Midnight

| Cosmos / CosmWasm Tool | Midnight Equivalent | Notes |
|---|---|---|
| Rust (contract language) | Compact (`compactc`) | TypeScript-like DSL; compiles to ZK circuits, not Wasm |
| `cargo build` / `optimizer` | `compactc` | Single compile step; produces ZK circuits + TS stubs |
| `cosmwasm-check` | `compactc` (strict compiler) | Errors on `disclose()` violations and circuit bound issues |
| `cw-storage-plus` | Built-in `export ledger` fields | No separate storage crate; declare fields directly |
| CosmJS / `@cosmjs/*` | Midnight.js SDK | TypeScript SDK for tx building and submission |
| CosmWasm schema (JSON schema) | Compiler-generated TS stubs | Typed circuit interfaces, auto-generated by `compactc` |
| `wasmd` / chain node | Midnight DevNet | Local chain for development and testing |
| `cw-multi-test` | Compact test suite | Tests run against local DevNet node |
| Keplr / Leap wallet | Lace wallet (Midnight-enabled) | Manages NIGHT, DUST, shielded UTXOs |
| Osmosis / Neutron deployment | Midnight testnet / mainnet | Deploy directly to Midnight; no chain choice needed |
| *(no equivalent)* | Proof server daemon | Separate process for client-side ZK proof generation |
| `@openzeppelin/compact-contracts` | Audited contract modules | `Ownable`, `Pausable`, `FungibleToken` — experimental |
| Midnight MCP | AI-assisted dev tool | Live Compact compiler integration for AI editors |

### Project Structure

```
my-midnight-dapp/
├── contracts/
│   ├── my_contract.compact          # Compact source (replaces src/lib.rs + contract.rs)
│   └── managed/                     # Compiler output — do not edit
│       └── my_contract/
│           ├── contract/index.cjs   # TypeScript stubs (replaces schema JSON)
│           ├── keys/                # ZK proving/verifying keys
│           └── zkir/                # ZK intermediate representation
└── src/
    ├── index.ts                     # dApp logic via Midnight.js SDK
    └── witnesses.ts                 # Implement witness functions here
                                     # (secret key access, off-chain data)
```

### Development Workflow

1. **Write contract:** Author `.compact` file. Declare ledger fields, witness function signatures, exported circuits.
2. **Compile:** Run `compactc`. The compiler is strict: `disclose()` violations and non-constant loop bounds are compile errors, not runtime warnings.
3. **Start proof server:** Run the proof server daemon. Your dApp's SDK calls route through it for proof generation. Budget for this process in dev setup and deployment architecture.
4. **Implement witnesses:** Compiler generates TypeScript interfaces for each witness function. You implement the bodies — secret key retrieval, off-chain data queries.
5. **Test:** Run against local Midnight DevNet. Test circuit correctness, state transitions, proof generation, and both happy paths and failing assertions.
6. **Deploy:** Midnight.js SDK. Deployer's wallet pays DUST. Contract is immutable post-deployment.
7. **Build dApp:** Use compiler-generated TypeScript stubs as typed circuit interfaces. Submit transactions via the SDK.

## 11. Security: What Changes, What Disappears, What's New

CosmWasm has a well-documented set of security considerations: missing sender authentication, reentrancy through submessages (rare but possible given CosmWasm's wasmvm vulnerabilities in 2025), integer overflow, unauthorized admin transfers, and the subtleties of CW-20 allowance management. Midnight eliminates several structurally while introducing a small, distinct set of new concerns.

### CosmWasm Vulnerabilities That Don't Apply on Midnight

- **Reentrancy via submessages:** Compact circuits are closed computations. No external calls possible mid-execution. No reentrancy vector exists.
- **Missing sender checks:** Replaced structurally. There is no `info.sender` to forget checking. If you omit the `assert`, the circuit simply has no access control — visible at audit time.
- **CW-20 allowance bugs:** The approve/allowance model doesn't exist in Midnight's UTXO token model. Entire class of allowance manipulation bugs is absent.
- **Contract migration vulnerabilities:** No `migrate` entry point; no in-place upgrade. Logic is fixed at deployment.
- **Storage key collisions:** No manual storage key management. Ledger fields are typed and compiler-managed.
- **Front-running via mempool:** Shielded transactions don't expose amounts or intent pre-broadcast. No MEV surface for value extraction.

### Security Patterns That Still Apply

- **`assert()` for invariants:** Same role as CosmWasm's `StdError` returns. Proof fails on violation — transaction rejected cleanly.
- **Arithmetic bounds:** Compact uses fixed-width integers (`Uint<64>`, `Uint<128>`). No arbitrary-precision integers. Audit overflow and underflow for all arithmetic.
- **Access control completeness:** Every restricted `export circuit` needs its own `assert`. No shared context or inherited ownership. Review every circuit.
- **Round counter discipline:** Always include a `round: Counter` field and call `round.increment(1)` in every circuit that modifies state. This is Midnight's equivalent of a nonce — prevents transaction correlation and replay patterns.

### New Attack Surfaces

**`ownPublicKey()` / naive public key comparisons** are the most dangerous footgun for CosmWasm devs moving to Midnight. As covered in §5, calling a stdlib function to retrieve the user's public key inside a circuit does not cryptographically prove key ownership — it compiles to an unconstrained input. Always use the witness + helper circuit + `persistentHash` pattern with domain separation. This was confirmed as a live vulnerability in the OZ Compact `Ownable` module in April 2026. If you consume library code, audit the access control pattern.

**Disclosure mistakes** are the most common Compact bug class. Using `disclose()` on sensitive data you didn't intend to make public, or storing private-origin values in ledger state without realizing what's been exposed. The compiler catches accidental leakage of witness data — but can't catch *intentional* disclosure you didn't think through. Audit every `disclose()` call.

**Commitment brute-forcing:** Committing to a small-domain value (boolean, small integer) without a random salt makes the commitment trivially invertible. Use `persistentCommit()` with randomness for small-domain secrets.

**Witness implementation bugs:** `compactc` validates the circuit interface and data flow. It cannot validate your TypeScript witness implementations. A buggy witness can leak data or return stale values without compile-time detection. Test witness logic independently. The `@openzeppelin/compact-contracts` library ships test witnesses only — not production witnesses. If you use it, author and audit your own.

**Partial-success states:** Given the two-phase execution model (§9), a fallible-phase failure can leave your contract in a partially-applied state with no clean revert. Design circuit boundaries so partial success is always safe.

**Circuit constraint complexity:** Very large circuits produce long proving times. Keep circuits lean and bounded — the constraint count at compile time is your cost model.

## 12. Quick Reference: CosmWasm / Cosmos → Compact / Midnight

| CosmWasm / Cosmos | Compact / Midnight | Notes |
|---|---|---|
| `instantiate(deps, env, info, msg)` | `constructor()` | No `info`; no `deps`; sets ledger fields directly |
| `execute(deps, env, info, msg)` | `export circuit myFn(params)` | `export` = callable in a transaction |
| `query(deps, env, msg)` | Read `ledger.field` via SDK | No on-chain query entry point |
| `migrate(deps, env, msg)` | Redeploy new contract | No in-place migration |
| `reply(deps, env, msg)` | No equivalent | No submessage callbacks |
| `sudo(env, msg)` | No equivalent | No privileged module entry point |
| `ibc_*` entry points | No equivalent (roadmap) | IBC not available in current Midnight |
| `info.sender` | Custom `persistentHash`-based witness pattern | See §5 — naive `publicKey()` is a security vulnerability |
| `info.funds` | No equivalent | No inline coin transfer with message |
| `env.block.time` | `round: Counter` in ledger | No timestamps; use round for ordering |
| `env.contract.address` | Contract address (off-chain) | Available via SDK, not in circuit |
| `deps.storage` (`Item` / `Map`) | `export ledger field: Type;` | Typed fields; no key-value API |
| `item.save(storage, &v)?` | `ledger.field = disclose(v)` | `disclose()` required for private-origin values |
| `item.load(storage)?` | `ledger.field` | Direct access in circuit |
| `StdError::generic_err("msg")` | `assert(false, "msg")` | Proof fails; transaction rejected |
| `CosmosMsg::Bank::Send` | Zswap UTXO output | UTXO model replaces bank sends |
| CW-20 transfer | Zswap UTXO or account-style circuit | Choose model per use case |
| CW-20 `increase_allowance` | No equivalent | ZK proof owns the spend; no allowance needed |
| `CosmosMsg::Wasm(Execute{..})` | **Not implemented in Compact 1.0** | Runtime cross-contract calls unavailable |
| `QueryRequest` (`deps.querier`) | No equivalent | Read off-chain via SDK; no on-chain query |
| Gas / ATOM fees | DUST fees | Generated by holding NIGHT; non-traded |
| `IBC CosmosMsg::Ibc` | No equivalent | Not available in current Midnight |
| `cw-ownable` | Custom witness + `persistentHash` circuit | See §5 — don't use unconstrained public key patterns |
| `.add_attribute()` / `.add_event()` | No events | Observe ledger state changes instead |
| `#[cfg(test)]` + `cw-multi-test` | Compact test suite + DevNet | Tests against local DevNet node |
| `cargo build --target wasm32` | `compactc` | Compiles to ZK circuits + TS stubs |
| CosmJS | Midnight.js SDK | TypeScript off-chain SDK |
| Keplr wallet | Lace wallet | Manages NIGHT, DUST, shielded state |
| `u64` / `u128` | `Uint<64>` / `Uint<128>` | Fixed-width integers; explicit size |
| `Addr` / `String` | `Bytes<32>` (key commitment) | Addresses are key commitments in Compact |
| `bool` | `Boolean` | Same concept |
| `keccak256` | `persistentHash()` / `transientHash()` | ZK-optimized hash functions; always use `persistent*` for ledger state |
| Commitment with salt | `persistentCommit(v, rand)` | Always use a random salt for small-domain values |

## 13. Suggested Learning Path

1. **Install the toolchain** → start the proof server → clone `example-hello-world` → deploy to local DevNet. Get the compile → proving-keys → TypeScript-driver loop in your fingers before writing real contract logic. ([docs.midnight.network/getting-started](https://docs.midnight.network/getting-started))
2. **Work the bulletin-board tutorial.** It introduces witnesses, the correct `publicKey` / `persistentHash` identity commitment pattern, `disclose()`, and the round-counter replay pattern in a realistic contract. Study it closely — the access control pattern there is the right one.
3. **Read the smart contract security page** ([docs.midnight.network/compact/smart-contract-security](https://docs.midnight.network/compact/smart-contract-security)). Specifically the section on hash-based authentication and the warning about small-domain commitment brute-forcing.
4. **Read the Compact language reference and standard library** ([docs.midnight.network/compact](https://docs.midnight.network/compact)). Pay attention to ledger types: `Counter`, `Map`, `MerkleTree`, `Maybe`.
5. **Read the "How Midnight Works" concept pages** — transaction semantics (especially guaranteed/fallible phases), Zswap, DUST architecture, Kachina.
6. **Build something with a native token** to feel the Zswap UXTXO model, then add selective disclosure to feel the compliance angle.
7. **Explore `@openzeppelin/compact-contracts`** for `Ownable`, `Pausable`, and `FungibleToken` patterns — actively maintained but flagged experimental. Author your own witnesses; do not reuse test witnesses from the library.
