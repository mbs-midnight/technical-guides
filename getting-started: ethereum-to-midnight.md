# From Ethereum to Midnight — Developer Field Guide

*Everything you know is useful. Some of it needs updating.*

**Midnight Network Foundation | June 2026**

> **Version note:** Midnight mainnet launched March 2026. Compact targets `language_version >= 0.23`. Low-level Zswap/coin mechanics are flagged by the official docs as not yet fully stable — always verify against [docs.midnight.network](https://docs.midnight.network) before shipping.

---

## 1. Why Midnight? The Pitch for Ethereum Devs

You've been building on Ethereum — or an EVM-compatible chain — where every transaction, every address balance, and every contract call is public by default. That's great for auditability. It's terrible for anything involving sensitive business logic, personal data, or compliance requirements where you can't just dump everyone's financials on a globally readable ledger.

Midnight flips that default. Privacy is baked into the protocol using zero-knowledge proofs, not bolted on as an afterthought. Data stays off-chain unless you explicitly choose to disclose it. On-chain, validators see only cryptographic commitments and proofs — enough to verify correctness, nothing more.

**One-line pitch:** Midnight is a privacy-first, UTXO-based blockchain where smart contracts are proved correct off-chain with ZK proofs, not re-executed on-chain by every node.

### What Carries Over from Ethereum

- Smart contracts with on-chain state, entry points, and composability
- TypeScript-like syntax in the smart contract language (Compact)
- A JavaScript/TypeScript SDK for off-chain dApp logic
- Token standards, governance patterns, access control concepts
- OpenZeppelin-style audited libraries (the `@openzeppelin/compact-contracts` package exists and is actively maintained — flag it as experimental for now)

### What Doesn't Carry Over

- `msg.sender`, `msg.value`, `block.timestamp` — Midnight doesn't expose these
- The EVM account model — Midnight uses UTXOs
- On-chain execution — circuit logic runs off-chain; only the proof hits the chain
- Solidity — Compact is a different language with a genuinely different mental model
- Gas paid in a volatile native token — fees are paid in DUST, a non-transferable resource generated by holding NIGHT

## 2. The One Mental Shift That Matters Most

If you internalize nothing else, internalize this:

**On Ethereum, the chain executes your contract. On Midnight, the chain verifies a proof that you executed it correctly.**

On Ethereum, every validator re-runs your bytecode in the EVM. State is global, computation is public, and "the network agrees on the result" because everyone redid the math. Privacy is impossible by construction — transparency is the consensus mechanism.

On Midnight, your contract logic runs **off-chain, on the user's machine**. That execution produces a zero-knowledge proof. The chain receives the proof plus a declared state transition, verifies the proof, and applies the transition. Validators never see your inputs and never re-execute your logic.

> **Analogy:** Ethereum is a classroom where everyone shows their work on the whiteboard and the whole class redoes the arithmetic to agree you got it right. Midnight is a classroom where you do the work privately, hand in a tamper-proof sealed envelope that mathematically guarantees you followed the rules, and the teacher checks the seal. The teacher learns *that you were right* without learning *what you did*.

Everything downstream in this guide — privacy-by-default, the three-part contract structure, witnesses, the token model — falls out of that one idea.

## 3. The State Model: Accounts → UTXOs

This is the biggest structural change for Ethereum developers. Ethereum runs an **account model** — there's a global state trie where every address has a balance. Midnight runs a **UTXO model**, extended with ZK-native privacy features.

### Account Model (Ethereum)

Think of it like a bank account. You have one balance. Transactions are state updates: `Alice.balance -= 50`, `Bob.balance += 50`. Contracts also have accounts. Everything lives in a global mutable state.

```solidity
mapping(address => uint256) public balances;

function transfer(address to, uint256 amount) external {
    require(balances[msg.sender] >= amount);
    balances[msg.sender] -= amount;
    balances[to] += amount;
}
```

### UTXO Model (Midnight)

Think of it like physical cash. You hold discrete coins (UTXOs). To spend one, you consume it entirely and create new outputs — one for the recipient, change back to yourself. There's no "balance update"; only creation and consumption of immutable pieces of value.

```
// Conceptual — not Compact syntax
AliceWallet = [UTXO_1: 100 NIGHT, UTXO_2: 50 NIGHT]

// To send 60 NIGHT:
Transaction {
  inputs:  [UTXO_1]                           // consume 100 NIGHT entirely
  outputs: [{ value: 60, owner: Bob },         // payment
            { value: 40, owner: Alice }]        // change back to self
}
// UTXO_1 is gone. Two new UTXOs exist.
```

### Why UTXOs Enable Privacy

In an account model, all activity links back to your address — a growing ledger of your financial history. UTXOs are naturally isolated. More importantly, Midnight's UTXOs can be **shielded**: value and owner are hidden behind cryptographic commitments. The network knows a UTXO exists; it doesn't know how much it contains or who owns it.

Double-spend prevention on shielded UTXOs uses a **nullifier set**. When you spend a UTXO, you publish a nullifier (a hash derived from the UTXO and your private key). The network checks the nullifier isn't already in the set — without learning which UTXO was spent.

| Aspect | Ethereum | Midnight |
|---|---|---|
| State model | Account / global trie | UTXO set + nullifier set |
| Balance query | `accounts[addr]` | Sum unspent UTXOs in wallet |
| Transaction effect | Mutates global state | Consumes inputs, creates outputs |
| Privacy default | Everything public | Everything private unless disclosed |
| Sender identity | `msg.sender` always available | Not exposed to contracts |
| Value in-flight | `msg.value` | Use Zswap / token UTXOs |
| Double-spend guard | Nonce increment | Nullifier set membership proof |
| Parallelism | Sequential (nonce ordering) | Natural (independent UTXOs) |

## 4. Tokens: NIGHT, DUST, and Custom Tokens

### NIGHT — The Network Token

NIGHT is the transferable utility token. It's used for staking, governance, and generating DUST. Think of it like ETH — it's the thing you hold to participate in the network — but it is **not** directly spent to pay transaction fees.

- Staked by validators to secure the network
- Tradeable on exchanges
- Used for governance
- Generates DUST passively for the holder
- All NIGHT transfers are **publicly visible on-chain** — it is deliberately not a privacy coin

### DUST — The Fee Resource

DUST is the non-transferable resource used to pay transaction fees. It's generated continuously by holding NIGHT. This is the part that feels weirdest until it clicks.

> **Analogy:** ETH gas is paying for your phone in cash every call — and the cash price swings daily. DUST is a phone plan where owning the handset (NIGHT) entitles you to a monthly bucket of minutes (DUST) that refills on its own. You can't sell your minutes to a friend, and unused minutes expire — but you always know roughly how many calls you can make by looking at your plan tier.

Key properties:
- **Shielded.** Using DUST doesn't expose your wallet address. Fee activity is private.
- **Non-transferable.** You cannot send DUST to another wallet. It can be *delegated* to sponsor others' transactions, but not transferred outright.
- **Decaying.** Unused DUST bleeds off. If you move your NIGHT or un-designate the generating address, DUST decays to zero.
- **Renewable.** Consume it on a transaction and it refills over time based on your NIGHT holdings.

**Why this design is clever:** it decouples *cost of using the network* from *price of the network's token*. A dApp paying fees in DUST doesn't care if NIGHT's market price triples overnight — its transaction capacity is a function of how much NIGHT it holds, not what NIGHT is worth. For enterprises modeling predictable infrastructure costs, that's the whole ballgame.

**Sponsored transactions:** A dApp can hold NIGHT, generate DUST, and delegate that DUST to cover users' fees — making the app feel "gasless" at the point of use. This is conceptually similar to ERC-4337 paymasters but built into the base resource model, not a contract layer on top.

### Custom Tokens (Shielded and Unshielded)

Custom tokens on Midnight are represented as UTXOs, not as entries in a contract's `mapping`. A token type is a **256-bit identifier** derived from a hash of the issuing contract's address plus a developer-chosen domain separator. The native token (NIGHT) uses the reserved zero value.

This means token identity is cryptographically bound to its issuing contract — no two contracts can accidentally or maliciously claim the same token type. No global registry needed.

| Property | ERC-20 (Ethereum) | Custom Token (Midnight) |
|---|---|---|
| Storage | `mapping` in contract state | UTXOs on the ledger |
| Transfer | `balanceOf` mutation | Consume input UTXO, create output UTXO |
| Privacy | None natively | Shielded UTXOs available |
| Token identity | Contract address | Hash(contractAddress, domainSeparator) |
| Spend authorization | `msg.sender` check | ZK proof of ownership |
| Atomic multi-asset swap | Complex, multiple calls | Native via Zswap protocol |

## 5. Zswap: Shielded Token Transfers and Atomic Swaps

Zswap is Midnight's shielded token mechanism, based on **Zerocash** and extended with native multi-asset support and atomic swaps. It's not a DEX contract bolted on top — it's baked into how transactions work at the protocol level.

### Coins, Commitments, and Nullifiers

A coin isn't a balance entry. Its existence is recorded on-chain only as a **commitment** — a cryptographic hash that hides the coin's type, value, and owner. Commitments accumulate in a global **Merkle tree**.

To spend a coin, you don't point at its commitment (that would link spend to creation and destroy privacy). Instead you publish a **nullifier** — a unique, one-way tag derived from the coin — and a ZK proof that says: *"this nullifier corresponds to some commitment that exists in the Merkle tree, and I own it,"* without revealing which commitment.

- The **nullifier set** prevents double-spends: each coin can only be nullified once.
- The **unlinkability** between commitments and nullifiers is the privacy. Observers see coins being created and coins being destroyed, but can't match them up.

> **Analogy:** A coat-check issues numbered claim tickets (commitments) for every coat. To retrieve yours, instead of handing back your ticket number, you provide a sealed proof that says "I hold a valid ticket for some coat in this room, and it's mine" — and the attendant marks that coat as collected (the nullifier). No one learns which ticket was yours. Coats come and go, but arrivals can't be matched to departures.

### Offers: The Building Block of Value Movement

A Zswap **offer** is the unit of value transfer in a transaction. It has four parts:

1. **Inputs (spends)** — coins being consumed. Each carries a nullifier, a Pedersen commitment to its type/value, a Merkle root reference, and a ZK proof. Valid only if the proof verifies *and* the referenced Merkle root is a recognized past root.
2. **Outputs** — coins being created. Each carries a new commitment, a Pedersen commitment to type/value, an optional recipient ciphertext, and a ZK proof.
3. **Transient coins** — coins created and consumed within the *same* transaction. Useful when contract logic needs to move value through intermediate steps without touching the global UTXO set.
4. **Balance vector** — a per-token-type tally. Inputs count positive, outputs negative. An offer is **balanced** when every dimension is non-negative, after adjusting for mints and fees. This invariant replaces "sender balance >= amount."

**How atomic swaps work without an escrow contract:** two parties' offers are merged, and the transaction is only valid if the combined balance across all token types nets to zero. Value can't be created or destroyed, but it can change hands atomically across asset types — with individual amounts staying shielded.

**No `approve()` pattern.** Token spending authorization comes from ZK proofs of coin ownership. There's no separate allowance transaction. No mempool exposure, no front-running surface.

> **Zswap implementation note:** the official docs flag this layer as not yet performance-optimized and subject to further revision. The conceptual model is stable; specific API surface and proof costs should be verified against current docs before building production integrations.

## 6. Compact vs. Solidity: The Contract Language

### How Contracts Execute

In Ethereum, your Solidity contract runs on every validating node. The EVM re-executes the code to verify correctness. Everyone sees everything.

In Midnight, your Compact contract **compiles to ZK circuits**. When a user invokes a circuit, it runs locally on their machine, consuming private inputs (witnesses). The result is a cryptographic proof that the computation was done correctly. That proof — not the computation — is submitted to the chain. Validators verify the proof; they never see the private data.

### The Three Contexts

A Compact contract has three distinct contexts — this is the first real structural difference from Solidity:

| Context | What it is | Ethereum analog |
|---|---|---|
| **Ledger** | Public, on-chain state. Visible to all. | Storage variables |
| **Circuits** | Provable logic. Compiles to ZK circuits. Runs off-chain to produce a proof. | Function bodies — but proven, not re-executed on-chain |
| **Witnesses** | Private off-chain data and computation. Implemented in TypeScript. Only the executing user knows it. | No equivalent. Genuinely new. |

### Basic Structure

```compact
pragma language_version >= 0.23;
import CompactStandardLibrary;

// On-chain state — public, persistent, visible to all
export ledger round: Counter;
export ledger owner: Bytes<32>;

// Witness: private, implemented in TypeScript, never on-chain
witness getSecretKey(): Bytes<32>;

// Entry point — exported = callable in a transaction
export circuit increment(): [] {
    round.increment(1);
}
```

Each ledger field is declared individually with `export ledger fieldName: Type;`. There is no block syntax.

### Solidity vs. Compact — Keyword Translation

| Solidity | Compact | Notes |
|---|---|---|
| `pragma solidity ^0.8.0;` | `pragma language_version >= 0.23;` | Version-pins the language |
| `import "@oz/...";` | `import CompactStandardLibrary;` | Stdlib must be imported explicitly |
| `uint256 public x;` | `export ledger x: Uint<64>;` | Ledger = public on-chain state |
| `mapping(addr => uint)` | `Map` (stdlib), or off-chain | Use Merkle trees for privacy-preserving maps |
| `function f() external` | `export circuit f()` | `export` = callable entry point |
| `function f() internal` | `circuit f()` | No `export` = internal helper |
| `require(cond, "msg")` | `assert(cond, "msg")` | Proof fails on violation |
| `msg.sender` | No equivalent | See below |
| `msg.value` | No equivalent | Use Zswap token UTXO flows |
| `emit Event(...)` | No events | Read ledger state changes |
| `immutable` / `constant` | `sealed` ledger field | Set only at deploy time, immutable after |
| `bytes32` | `Bytes<32>` | Fixed-size byte arrays |
| `uint64` | `Uint<64>` | Explicit bit-width |
| `keccak256(abi.encode(x))` | `transientCommit(x, salt)` or `persistentCommit(x, rand)` | ZK-native commitment schemes |
| `constructor()` | `constructor()` | Same concept, different syntax |
| `modifier onlyOwner` | `assert(publicKey(sk()) == owner)` inline in each circuit | No modifiers; pattern is repeated |
| `selfdestruct` / `delegatecall` | No equivalent | Contracts are immutable once deployed |

### Side-by-Side: A Stateful Contract with Access Control

**Solidity:**

```solidity
pragma solidity ^0.8.0;

contract Vault {
    address public owner;
    uint64 public value;
    bool public isSet;

    constructor() { owner = msg.sender; }

    function set(uint64 newValue) external {
        require(!isSet, "Already set");
        require(msg.sender == owner, "Not owner");
        value = newValue;
        isSet = true;
    }

    function clear() external {
        require(isSet, "Nothing to clear");
        require(msg.sender == owner, "Not owner");
        isSet = false;
    }
}
```

**Compact:**

```compact
pragma language_version >= 0.23;
import CompactStandardLibrary;

enum State { UNSET, SET }

export ledger authority: Bytes<32>;
export ledger value: Uint<64>;
export ledger state: State;
export ledger round: Counter;  // anonymity counter — see security section

witness getSecretKey(): Bytes<32>;

constructor(sk: Bytes<32>, initialValue: Uint<64>) {
    authority = disclose(persistentHash(sk));
    value = disclose(initialValue);
    state = State.UNSET;
}

export circuit set(newValue: Uint<64>): [] {
    assert(state == State.UNSET, "Already set");
    // Prove caller knows the secret behind authority — without revealing the secret
    assert(authority == persistentHash(getSecretKey()), "Not authorized");
    value = disclose(newValue);
    state = State.SET;
    round.increment(1);
}

export circuit clear(): [] {
    assert(state == State.SET, "Nothing to clear");
    assert(authority == persistentHash(getSecretKey()), "Not authorized");
    state = State.UNSET;
    round.increment(1);
}
```

Look at `clear()`. There's no `require(msg.sender == owner)`. The user feeds their secret key in through a witness, the circuit hashes it and compares it against the stored `authority` — and the proof attests "the person who built this transaction knows the secret behind `authority`" without the key ever touching the chain. That's the paradigm in one function.

### Where's `msg.sender`?

There isn't one. On Ethereum, the sender is ambient context the EVM injects, and it's publicly knowable. On Midnight, transactions are unlinkable and private by default, so there's no public "from" address to read. **You bring your own identity as a witness and prove whatever you need about it in-circuit.** Authorization becomes a cryptographic statement, not a runtime variable.

Practical consequence: `onlyOwner` via `msg.sender`, address allowlists, `tx.origin` — all get re-expressed as commitments, hashes, and ZK proofs.

### Privacy Is the Default, Not an Opt-In

In Compact, **circuit inputs are private by default**. The compiler tracks data flow. Anything derived from a witness is considered private. If you try to store it on-chain without explicitly wrapping it in `disclose()`, **the compiler rejects it at compile time**.

```compact
witness _privateAmount(): Uint<64>;

export circuit transfer(recipient: Bytes<32>): [] {
    const amount = _privateAmount();

    // ❌ COMPILE ERROR: cannot place witness-derived value on-chain
    ledger.total = ledger.total + amount;

    // ✅ CORRECT: explicitly mark this as safe to disclose
    ledger.total = ledger.total + disclose(amount);
}
```

`disclose()` is a compiler signal that says "I know this value is leaving the private domain; I accept responsibility." The value only appears on-chain if it's actually stored in ledger state. The pattern enforces intentional disclosure — you can't accidentally leak private data.

The mental model inverts from Ethereum: on Ethereum you fight to hide things; on Midnight you fight to reveal exactly the right things.

### Bounded by Construction

EVM execution is metered at runtime — you can run out of gas halfway through a loop. Compact circuits are **bounded at compile time**. Loops have fixed bounds, computation has a known ceiling, the compiler enforces it.

```compact
// ✅ Bounded — compile-time constant ceiling
for (let i = 0; i < 10; i++) {
    if (i < count) { total = total + ledger.items[i]; }
}

// ❌ Not valid — no dynamic loop bounds
while (i < dynamicArray.length) { ... }
```

Upside: predictable proving cost, no runtime gas-exhaustion class of bugs. Downside: no unbounded loops. Plan your data structures around fixed bounds.

## 7. Witnesses: The Part With No Ethereum Analog

A **witness** is a function *declared* in Compact but *implemented* in TypeScript on the user's machine. It supplies private data into a circuit — a secret key, an off-chain credential, a private balance — without that data ever going on-chain or being seen by anyone else.

```compact
// Compact — declaration only
witness getSpendingKey(): Bytes<32>;
```

```typescript
// TypeScript — actual implementation
const witnesses = {
    getSpendingKey: () => wallet.getPrivateKey()
};
```

**Critical security rule:** witness implementations run off-chain and are **not cryptographically verified**. Each user supplies their own. Your contract logic must never *trust* a witness value without validating it inside the circuit. A witness is "here's a value I claim" — the circuit's job is to assert the claim is consistent with public state before acting on it. Treat every witness as untrusted user input, because that's exactly what it is.

A buggy or malicious witness can leak secrets, produce invalid proofs, or undermine the guarantees of the contract. OpenZeppelin's Compact library explicitly ships test witnesses only — not production witnesses — for exactly this reason. If you consume that library, you must author and audit your own witnesses for production use.

> **Analogy:** A witness is like showing a bouncer your ID. The bouncer (the circuit) checks that the ID is valid and the photo matches, then lets you in — but the club's guest list (the public ledger) never records your home address. The proof says "an authorized person entered," not "here is everything on their driver's license."

## 8. Transactions on Midnight

A Midnight transaction is richer than Ethereum's `{from, to, value, data, gas, nonce}`. It bundles:

- One or more **circuit invocations** with their ZK proofs
- **UTXO inputs** being consumed (including shielded token inputs)
- **UTXO outputs** being created (payments, change, new token UTXOs)
- **DUST payment** for fees
- **Nullifiers** for any shielded UTXOs being spent

### Public vs. Shielded

A single transaction can mix shielded and unshielded components:

| Property | Unshielded | Shielded |
|---|---|---|
| Value visible | Yes | No — hidden by commitment |
| Sender/receiver visible | Yes | No — hidden by ZK proof |
| On-chain footprint | UTXO with plaintext value | Commitment hash + nullifier |
| Proof required | Basic signature | Full ZK proof of ownership + balance |

### The Two-Phase Execution Model

Ethereum transactions are atomic: they fully succeed or fully revert. Midnight splits execution into phases, and the difference matters.

**Phase 1 — Well-formedness check:** Runs against no state. Verifies all ZK proofs, the Schnorr proof on the contract section, and that offers are balanced (accounting for fees and mints). A bad proof or unbalanced offer → rejected outright, not included.

**Phase 2 — Guaranteed phase:** Runs against ledger state. **Fees are collected here.** If this phase fails, the transaction is *not included in the ledger at all*.

**Phase 3 — Fallible phase:** Also runs against ledger state. If this fails, the effects of the guaranteed phase **still apply**, and the transaction is recorded as a **partial success**. Fees are forfeited.

> **Why this matters:** a transaction can land in a "partially applied" state with no clean revert. Design interactions so that a partial success is always a safe state to be in. Don't put an invariant-critical step in the fallible phase if the guaranteed phase already committed something that depends on it. Fee payment and must-succeed core logic belong in the guaranteed phase. Speculative or best-effort effects go in the fallible phase.

### Proving Happens Client-Side

Because execution and proof generation happen on the user's machine, **transaction construction is heavier on the client than on Ethereum**. Proof generation for typical circuits currently takes seconds. Budget for it in UX. A proof server can be run separately from your dApp to offload this work, and Midnight's toolchain supports it.

## 9. Compilation and Project Structure

The Compact compiler (`compactc`) translates your `.compact` files into:

```
managed/<contract>/
├── contract/   # Generated TypeScript API + JS implementation (your typed "ABI")
├── keys/       # Proving and verifying keys for the ZK circuits
├── zkir/       # Zero-Knowledge Intermediate Representation
└── compiler/   # JSON metadata about the contract
```

Your contract ships with a **typed TypeScript driver**. No hand-rolling ABI encoders — your dApp imports the generated module and calls circuits like typed async functions. The SDK handles proof generation under the hood.

### Toolchain Overview

| Tool | Purpose | Ethereum Equivalent |
|---|---|---|
| `compactc` | Compiles `.compact` → ZK circuits + TypeScript stubs | `solc` |
| Midnight.js SDK | TypeScript SDK for dApps and transaction submission | ethers.js / web3.js |
| `@openzeppelin/compact-contracts` | Audited token and governance modules (experimental) | OpenZeppelin Solidity |
| Midnight DevNet | Local development network | Hardhat / Anvil |
| Midnight MCP | AI-assisted Compact development with live compiler validation | Hardhat + Copilot (roughly) |

### Project Structure

```
my-dapp/
├── contracts/
│   ├── my_contract.compact           # Compact source
│   └── managed/
│       └── my_contract/              # Compiler output (generated)
│           ├── contract/index.cjs    # TypeScript stubs
│           └── keys/                 # ZK proving/verifying keys
└── src/
    ├── index.ts                      # dApp logic using Midnight.js SDK
    └── witnesses.ts                  # Implement witness functions here
```

### Compile-to-Deploy Workflow

1. Write your `.compact` contract
2. Run `compactc` to generate ZK circuits and TypeScript stubs
3. Implement witness functions in TypeScript (this is where private key logic lives)
4. Use the Midnight.js SDK to deploy the contract and submit transactions
5. Wallet provides spending key for DUST fee payment and proof generation

## 10. Security Patterns: Old Habits Die Hard

### What Still Works

- **`assert()` for invariants:** Same idea as Solidity's `require()`. Proof fails if the assertion is violated.
- **Access control via stored commitments:** Equivalent to `Ownable`, but proven via witness rather than `msg.sender`.
- **Bounded arithmetic:** Midnight uses fixed-size types (`Uint<64>`, `Bytes<32>`). Overflow is a compile-time concern.
- **Commitment schemes:** `transientCommit()` and `persistentCommit()` for hiding values with a salt.

### What Doesn't Exist

- **Reentrancy:** Not a concern in its classic form. Circuits are fixed-bound computations with no synchronous on-chain external calls mid-execution. The textbook reentrancy attack doesn't map over.
- **Front-running:** Shielded transactions are encrypted before broadcast. No pending transaction mempool visibility for amounts.
- **Flash loans:** The UTXO model doesn't support borrow-within-a-transaction the same way. Atomic execution still works.
- **`selfdestruct` / `delegatecall`:** No equivalent. Contracts are immutable once deployed; state transitions always produce new output state.

### New Attack Surfaces

**Disclosure mistakes** are the most common Compact security bug. `disclose()` on data you didn't mean to make public. Always audit every `disclose()` call. The compiler catches accidental leakage of witness data but can't catch *intentional* disclosure you didn't think through.

**Commitment brute-forcing:** If you commit to a small-domain value (like a boolean or a small integer), use a random salt or switch to `persistentCommit()` with randomness. Without it, an adversary can enumerate the domain.

**Round-counter omission:** Without a round counter in ledger state, transaction correlation across calls becomes easier. Always include a `round: Counter` field and increment it in every circuit that modifies state. The bulletin-board tutorial shows this pattern.

**Untrusted witness values:** The contract defines the interface; you implement the logic in TypeScript. A buggy witness implementation won't be caught by the contract compiler. Validate witness-derived values in-circuit before acting on them. This is the new top-risk category — treat it with the same seriousness you'd treat untrusted calldata in Solidity.

**Partial-success states:** Make sure a fallible-phase failure can't strand your contract in an exploitable in-between state. Design your circuit boundaries accordingly.

## 11. Cheat Sheet

| Concept | Ethereum | Midnight |
|---|---|---|
| Execution model | On-chain, every node re-runs EVM | Off-chain, chain verifies ZK proof |
| State model | Global account trie | Public ledger (UTXO set + contract state) |
| Privacy default | Everything public | Shielded by default; reveal via `disclose()` |
| Language | Solidity → EVM bytecode | Compact → ZK circuits + TypeScript driver |
| Function | `function` | `circuit` |
| State variable | storage variable | `export ledger fieldName: Type;` |
| Entry point | `external` / `public` | `export circuit` |
| Caller identity | `msg.sender` (ambient, public) | Bring-your-own via witness, proven in-circuit |
| Auth pattern | `require(msg.sender == owner)` | `assert(hash(witnessKey()) == authority)` |

| Top threat | Reentrancy | Untrusted witnesses + over-disclosure |

## 12. Suggested Learning Path

1. **Install the toolchain** → clone `example-hello-world` → deploy to a local devnet. Get the compile → proving-keys → TypeScript-driver loop in your fingers. ([docs.midnight.network/getting-started](https://docs.midnight.network/getting-started))
2. **Work the bulletin-board tutorial.** It covers witnesses, commitments, `disclose()`, and the round-counter replay-prevention pattern in a realistic contract.
3. **Read the Compact language reference and standard library** ([docs.midnight.network/compact](https://docs.midnight.network/compact)). Pay particular attention to ledger types: `Counter`, `Map`, `MerkleTree`, `Maybe`.
4. **Read the "How Midnight Works" concept pages** — transaction semantics, Zswap, DUST architecture, Kachina. This is where protocol-level reasoning lives.
5. **Build something with a native token** to feel the Zswap UTXO model, then add selective disclosure to feel the compliance angle.
